# DDDとクリーンアーキテクチャの実装と検証

## 概要

このスキルは、ドメイン駆動設計（DDD）原則に従った実装がなされているかを検証し、クリーンアーキテクチャのレイヤー設計、ドメインモデリング、依存関係管理を行います。

## 入力形式

スキル呼び出し時に検証対象を指定できます:

```
/ddd-check
```

または特定のファイル/ディレクトリのみ検証:

```
/ddd-check --domain-only
/ddd-check --file <target_file>
```

## 実行プロセス

### ステップ1: ユビキタス言語の整合性チェック

**検証内容**:
- 定義された用語がコード内で正しく使用されているかを確認します。

**確認ファイル**:
```bash
# ユビキタス言語定義を読む
<path_to_ubiquitous_language_definition>

# コードで使用されている用語を確認
<path_to_code>
```

**チェック項目**:
- コアドメイン用語の使用
- サポートドメイン用語の使用

### ステップ2: レイヤー分離の検証

**ディレクトリ構造チェック**:
```
<project_root>/
├── domain/              # ドメイン層（ビジネスロジック）
├── application/         # アプリケーション層（ユースケース）
├── infrastructure/      # インフラ層（外部システム連携）
└── api/                 # API層（ハンドラー）
```

**検証ルール**:
- 依存関係の方向を確認し、許可される依存と禁止される依存を明確にします。

### ステップ3: エンティティと値オブジェクトの検証

**エンティティのルール**:
- 識別子を持ち、ライフサイクルがあることを確認します。

**値オブジェクトのルール**:
- イミュータブルであり、識別子を持たないことを確認します。

### ステップ4: 集約の境界検証

**集約のルール**:
- 集約ルートを通じてのみ内部エンティティにアクセスすることを確認します。

### ステップ5: ドメインサービスの検証

**ドメインサービスの条件**:
- ステートレスであり、複数のエンティティにまたがるロジックを持つことを確認します。

### ステップ6: ポート・アダプターパターンの検証

**ポート（インターフェース）の検証**:
- 正しいポート定義とアダプター実装を確認します。

## 出力形式

### 全チェック通過時

```
✅ DDD設計原則チェック完了

検証項目:
- [✅] ユビキタス言語の整合性
- [✅] レイヤー分離
- [✅] エンティティと値オブジェクトの区別
- [✅] 集約の境界
- [✅] ドメインサービスの適切性
- [✅] ポート・アダプターパターン

🎉 DDD原則に従った実装がなされています。
```

### 違反検出時

```
🔴 DDD設計原則違反を検出

検証項目:
- [✅] ユビキタス言語の整合性
- [🔴] レイヤー分離 - 1件の違反
- [✅] エンティティと値オブジェクトの区別
- [⚠️] 集約の境界 - 1件の警告

次のアクション:
- [ ] 違反の修正
- [ ] 警告の確認（必要に応じて修正）
- [ ] 再検証: /ddd-check
```
