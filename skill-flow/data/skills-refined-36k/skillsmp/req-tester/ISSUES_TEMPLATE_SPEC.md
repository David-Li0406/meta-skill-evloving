## 🔖 字段定义

### 文档头部（必填）

| 字段 | 格式 | 说明 | 示例 |
|------|------|------|------|
| 需求名称 | `> **需求名称**: [文本]` | 从需求文档提取的标题 | `> **需求名称**: 链路漏斗增加毛利` |
| 生成时间 | `> **生成时间**: YYYY-MM-DD HH:mm` | 问题清单生成时间 | `> **生成时间**: 2026-01-20 14:30` |
| 问题总数 | `> **问题总数**: N 个` | 总问题数量 | `> **问题总数**: 38 个` |
| 优先级分布 | `> **优先级分布**: P0: X \| P1: Y \| P2: Z` | 各优先级数量统计 | `> **优先级分布**: P0: 12 \| P1: 20 \| P2: 6` |

### 问题标题（必填）

**格式**: `## 问题 N [分类 · 优先级]`

**规则**:
- 编号从 1 开始连续递增
- 分类必须是以下之一：
  - `前因后果` - 业务背景、目标、价值
  - `边界定义` - 生效条件、适用范围、权限
  - `异常场景` - 失败处理、冲突解决、降级策略
  - `交互细节` - 文案、提示、样式、交互
  - `数据状态` - 字段定义、状态流转、数据约束
  - `性能兼容` - 响应时间、并发、浏览器兼容
- 优先级必须是：`critical` | `warning` | `info`

**示例**:
```markdown
## 问题 1 [前因后果 · critical]
## 问题 2 [边界定义 · warning]
## 问题 3 [异常场景 · critical]
```

### 字段 1: [位置] (必填)

**格式**: `**[位置]** 第X页 - 章节名称`

**规则**:
- 必须标注**明确的页码**或**章节名称**
- 便于产品快速定位原文

**示例**:
```markdown
**[位置]** 第1页 - 需求背景
**[位置]** 第3页 - 核心功能说明 - Pending 状态定义
**[位置]** 附录 - 数据字典
```

### 字段 2: [来源] (必填)

**格式**:
```markdown
**[来源]**
需求文档XX位置提到"原文引用"，但未说明：
- 疑问点1
- 疑问点2
- 疑问点3
```

**规则**:
- 必须引用**原文片段**（用引号包裹）
- 必须列出**具体疑问点**（至少2个）
- 疑问点使用无序列表（`-`）

**示例**:
```markdown
**[来源]**
需求文档第1页提到"协商中的状态均增加 Pending 属性"，但未说明：
- 为什么需要 Pending 状态？
- 现有流程有什么痛点？
- Pending 状态要解决什么问题？
```

### 字段 3: [测试视角] (必填)

**格式**:
```markdown
**[测试视角]**
作为测试人员，需要理解XX，才能：
1. 目的1
2. 目的2
3. 目的3
```

**规则**:
- 必须以"作为测试人员，需要理解XX，才能："开头
- 必须列出**至少2个测试目的**
- 使用有序列表（`1. 2. 3.`）

**示例**:
```markdown
**[测试视角]**
作为测试人员，需要理解业务背景和目标，才能：
1. 设计贴近真实业务场景的测试用例
2. 评估功能的优先级和测试范围
3. 理解什么是"测试通过"的验收标准
```

### 字段 4: [需澄清] (必填)

**格式**:
```markdown
**[需澄清]**
1. 分类1：具体问题1？
2. 分类2：具体问题2？
3. 分类3：具体问题3？
```

**规则**:
- 使用有序列表
- 每个问题必须以**分类标签**开头（如"业务背景："、"触发条件："）
- 每个问题必须以**问号**结尾
- 至少列出**2个问题**

**示例**:
```markdown
**[需澄清]**
1. 业务背景：当前运营团队在跟进协商时遇到什么困难？
2. 目标定义：Pending 状态要解决什么业务问题？
3. 成功标准：上线后如何判断功能达成了目标？
```

### 字段 5: [回答建议] (必填)

**格式**:
```markdown
**[回答建议]**
- 角度1：具体建议或示例
- 角度2：具体建议或示例
- 角度3：具体建议或示例
```

**规则**:
- 使用无序列表
- 每条建议必须**具体且可操作**
- 可以包含示例或模板
- 至少提供**2个角度**

**示例**:
```markdown
**[回答建议]**
- 用户故事：作为【运营人员】，我希望【系统自动标记超过7天未回复的协商】，以便【重点跟进减少流失】
- 数据支撑：当前有多少比例的协商因超时未跟进而流失？目标是降低到多少？
- 验收标准：如何判断这个功能达成了目标？（如：Pending 标记准确率 > 95%）
```

### 字段 6: [产品回答] (必填)

**格式**:
```markdown
**[产品回答]**
<!-- 在此填写 -->
```

**规则**:
- 必须保留注释 `<!-- 在此填写 -->`
- 产品填写时删除注释，直接填写答案
- 支持 Markdown 格式（列表、表格、代码块）
- 未回答的问题保留注释

**填写示例**:
```markdown
**[产品回答]**
业务背景：运营团队反馈，当前有约30%的协商因红人长时间未回复而流失。Pending 状态用于标记这些"协商中但超过7天未回复"的记录，便于运营重点跟进。

目标：降低协商流失率，从当前30%降至15%以下。

验收标准：
1. Pending 标记准确率 > 95%
2. 上线一个月后，协商流失率降至20%以下
```

### 问题分隔符（必填）

**格式**: `---`

**规则**:
- 每个问题结束后必须用 `---` 分隔
- 分隔符独占一行
- 最后一个问题后也需要分隔符

## 🔍 格式校验规则

生成 issues.md 后，必须通过以下校验：

### 必填项校验

- [ ] 文档头部：需求名称、生成时间、问题总数、优先级分布
- [ ] 每个问题必须包含 6 个字段：位置、来源、测试视角、需澄清、回答建议、产品回答
- [ ] 每个问题必须有分隔符 `---`

### 格式校验

- [ ] 问题编号连续（1, 2, 3...）
- [ ] 分类必须是 6 种之一
- [ ] 优先级必须是 critical/warning/info
- [ ] [来源] 必须有引号包裹的原文
- [ ] [测试视角] 必须以"作为测试人员"开头
- [ ] [需澄清] 必须是有序列表，问题以问号结尾
- [ ] [回答建议] 必须是无序列表
- [ ] [产品回答] 必须有注释占位符

### 内容质量校验

- [ ] [来源] 列出至少 2 个疑问点
- [ ] [测试视角] 列出至少 2 个测试目的
- [ ] [需澄清] 列出至少 2 个具体问题
- [ ] [回答建议] 提供至少 2 个角度

## 🎯 使用说明

### 产品填写指南

1. **定位问题**：通过 `[位置]` 字段找到需求文档对应页面
2. **理解背景**：阅读 `[来源]` 了解为什么要问这个问题
3. **明确诉求**：阅读 `[测试视角]` 理解测试人员的关注点
4. **针对回答**：根据 `[需澄清]` 列表逐条回答
5. **参考建议**：查看 `[回答建议]` 了解回答角度
6. **填写答案**：在 `[产品回答]` 删除注释后填写

### req-merge 解析规则

1. 读取 issues.md
2. 按 `---` 分割问题
3. 提取每个问题的 6 个字段
4. 检查 `[产品回答]` 是否有内容（不含注释）
5. 将回答整合到完备需求文档
