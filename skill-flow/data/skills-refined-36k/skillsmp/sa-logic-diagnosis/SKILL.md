# 系统化修订 (Systematic Revision)

## 概述

随机修改浪费时间并制造新的矛盾。快速修补掩盖了深层逻辑问题。

**核心原则：** 在尝试修改之前，始终找到逻辑根源。只治标不治本是失败的。

**违反此流程的字面意思就是违反学术修订的精神。**

## 铁律

```
没有根源调查，就不进行修改
(NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST)
```

如果你没有完成第一阶段，你就不能提出修改建议。

## 何时使用

用于任何写作问题：
- 逻辑验证失败
- 审稿人指出的矛盾
- 意外的论证走向
- 清晰度问题
- 格式/引用错误

**特别是在以下情况使用：**
- 时间紧迫（紧急情况让人想猜测）
- “只是一个小改动”看起来很明显
- 你已经尝试了多次修改
- 之前的修改无效
- 你不完全理解问题

## 四个阶段

你必须在进入下一阶段之前完成每个阶段。

### 第一阶段：逻辑根源调查

**在尝试任何修改之前：**

1. **仔细阅读反馈/问题**
   - 不要跳过审稿意见或自查发现的错误
   - 它们通常包含确切的线索
   - 完整阅读上下文
   - 记录具体章节、段落

2. **一致性重现**
   - 你能确定问题在哪里吗？
   - 具体的逻辑断点是什么？
   - 每次阅读都会产生歧义吗？
   - 如果不确定 → 收集更多反馈，不要猜测

3. **检查最近的修改**
   - 什么改变导致了这个问题？
   - 对比草稿版本
   - 新引入的论据、引用
   - 结构调整

4. **追踪逻辑流 (Trace Data Flow)**

   **当错误深埋在论证链中：**
   
   - 错误的结论源自哪里？
   - 什么前提导致了这个结论？
   - 向上追溯直到找到源头
   - 在源头修正，而不是在表现处修正

### 第二阶段：模式分析

**在修复之前找到模式：**

1. **寻找优秀范例**
   - 在同一篇论文或经典文献中找到处理类似逻辑的成功段落
   - 什么是有效的？

2. **对比参考**
   - 仔细阅读范例
   - 理解其逻辑结构

3. **识别差异**
   - 有效段落和问题段落有什么区别？
   - 列出每一个区别，无论多小

### 第三阶段：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清晰陈述：“我认为 X 是逻辑根源，因为 Y”
   - 写下来

2. **最小化测试**
   - 做最小的改动来测试假设
   - 一次只改一个变量

3. **在继续之前验证**
   - 逻辑通顺了吗？是 → 第四阶段
   - 没通？形成新假设
   - **不要**在上面叠加更多修改

### 第四阶段：实施修订

**修复根源，而非症状：**

1. **创建验证案例**
   - 最简单的重现问题的方式
   - 使用 `superpowers:sa-writing` 技能来写出正确的待证主张

2. **实施单一修订**
   - 解决识别出的根源
   - 一次一个改动
   - 不要夹带“顺便优化”

3. **验证修订**
   - 现在逻辑通顺了吗？
   - 没有破坏其他论点吗？
   - 问题真的解决了吗？

4. **如果修订无效**
   - 停止
   - 计数：你尝试了多少次修订？
   - 如果 < 3：回到第一阶段
   - **如果 ≥ 3：停止并质疑架构（见下文）**

5. **如果 3+ 次修订失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修订都在不同地方揭示新矛盾
   - 修订需要“大规模重写”
   - 每次修订都在别处制造新问题

   **停止并质疑基本面：**
   - 这个论证框架根本上是合理的吗？
   - 我们是因为惯性而坚持它吗？
   - 我们应该重构章节结构而不是修补句子吗？

   **在尝试更多修订之前与你的人类伙伴讨论**

## 危险信号 - 停止并遵循流程

如果你发现自己在想：
- “先快速改一下，以后再调查”
- “试着改改 X 看看行不行”
- “一次改好几个地方”
- “跳过验证，我手动看看”
- “可能是 X，我修一下”
- “我不完全理解但这样可能行”
- “再试一次”（已经试过 2+ 次）

**所有这些都意味着：停止。回到第一阶段。**
