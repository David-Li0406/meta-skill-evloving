## WHAT TO IMPLEMENT FOR REAL vs FAKE vs MOCK

### IMPLEMENT FOR REAL (Core Feature Logic)
**If you're testing it, implement it legitimately:**

- **Business Logic & Algorithms**: Core feature calculations, validation rules, workflow logic
- **Data Processing**: Transformations, filtering, sorting, aggregation
- **State Management**: Feature state, user interaction handlers, component lifecycle
- **Domain Models**: Data structures, entities, value objects with real behavior
- **Feature Workflows**: User journeys, process flows, decision trees

### FAKE INTERNALLY (Simple Working Versions)
**Use simple but real implementations for supporting infrastructure:**

- **Data Storage**: In-memory collections instead of databases
- **File Operations**: String/memory operations instead of disk I/O
- **Networking**: Hardcoded responses that match test scenarios
- **Configuration**: Simple constants instead of complex config systems
- **Caching**: Basic dictionaries instead of sophisticated cache layers

### MOCK ONLY TRUE EXTERNALS (Outside Your Control)
**Only mock dependencies completely external to this feature:**

- **Third-Party APIs**: External services you don't control
- **Hardware Interfaces**: Camera, sensors, GPS, device capabilities
- **System Services**: Push notifications, background processing, OS services
- **Future Features**: Components planned for later sprints/iterations

## GREEN Phase Implementation Strategy:

### 2. SERVICE IMPLEMENTATIONS (Business Logic Level)

### 4. SUPPORTING INFRASTRUCTURE (System Level)

- **Simple but real utilities** that actually work
- **Real dependency injection** setup
- **Fake external dependencies** with hardcoded but realistic responses
- **Real configuration** using simple constants

## Implementation Guidelines:

**MINIMAL BUT REAL CODE:**

- Write the simplest WORKING implementation that makes tests pass
- Use naive algorithms that are correct but unoptimized
- Hardcode realistic data/responses that satisfy test scenarios
- Implement real logic flow, even if basic
- NO shortcuts that bypass actual feature implementation

**SPEED STRATEGIES:**

- **Hardcode Smart**: Use real algorithms with hardcoded test data
- **In-Memory First**: Real operations on simple data structures
- **Naive Algorithms**: Correct but basic implementations (linear search vs binary search)
- **Progressive Complexity**: Add sophistication only when tests demand it
