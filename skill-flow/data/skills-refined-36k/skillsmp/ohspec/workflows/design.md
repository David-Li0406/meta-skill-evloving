# design 阶段 - 方案设计（需求化）

## 概述

**目标**：在“scan-of-record”证据基础上完成方案设计，输出 RFC §3-§5（契约/矩阵/DFX/决策）。  
默认不做全量扫描，仅在发现“证据缺口”时触发补扫。

**执行者**：架构设计师子代理（或编排器在 Codex 兼容模式下代行）

**输入**：
- `rfc.md`（已完成 §1-§2）
- `findings.json`（必须含 confirmed.key_files + confirmed.facts）
- `progress.json`（ASSESS 决策 + 状态机）

**输出**：
- `rfc.md`：补齐 §3-§5（需求化设计，尽量少写代码）
- `progress.json`：更新状态与门禁记录

---

## 执行流程

### 步骤 0：证据门禁（强制）

若 `findings.confirmed.key_files` / `findings.confirmed.facts` 未达标：
- 返回 `blocked`
- 给出“缺失角色/模块/关键词”的补扫建议
- 禁止直接写设计（避免“按模板想象”）

### 步骤 1：开发者模型 → 契约（顺序强制）

先明确“开发者心智模型”（谁用、如何用、误用预防、错误信息），再落到契约表与变更矩阵：
- §3.1 开发者模型与易用性（Public API）
- §3.2 接口契约表（需求化）
- §3.3 数据字典/字段契约（如适用）
- §3.4 对齐与变更矩阵（每条变更必须有 existing_ref 或 new_change_plan）

### 步骤 2：场景/边界/依赖影响矩阵（强制闭环）

必须覆盖：
- §3.5 场景规格（Gherkin：正常/异常/边界/不支持）
- §3.6 异常/边界矩阵（并发/时序/权限/多用户/重启等）
- §3.7 依赖影响矩阵（失败模式/回退/观测/影响范围）

### 步骤 3：DFX（量化）

在 §4 用 8 维度给出量化指标与验证方式；禁止“模糊描述”。

### 步骤 4：设计决策（可审计）

在 §5 给出：
- 备选方案（至少 1 个）
- 选定方案 + 取舍理由
- 风险/回滚/灰度
- 如 complexity=COMPLEX：必须 Mermaid 图示（交互流程或状态机）

### 步骤 5：状态更新

更新 `progress.json`：
- `state_machine.current_state`：DESIGNING（或推进到 PRECHECKING）
- `phases.design.status/completed_at`
- `audit_log` 追加 `design_completed`

---

## 注意事项

1. **模板锁定（强制）**：只能在既有 `rfc.md`（由 `templates/rfc.md` 初始化）里填充 §3-§5；禁止重排章节/另起目录/生成 TOC
2. **不写实现**：禁止完整函数/算法；最多 1-2 行调用示意（可选）
3. **不重复全扫**：仅“缺口触发”补扫，并记录补扫原因与范围
4. **可读性优先**：矩阵单元格保持短；长说明放条目详情清单
