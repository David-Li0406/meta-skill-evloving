## Phase 0: 代码库评估 (assess)

**目标**：评估代码库规模，决策最优扫描策略

**执行者**：编排器（自动执行，无需子代理）

**关键活动**：
1. 使用 Glob 统计文件数量和类型
2. 使用 Grep 预览关键词匹配数量
3. 估算上下文大小（文件数 × 平均行数）
4. 根据规模决策扫描策略

**输出**：扫描策略建议（写入 progress.json）

> 注意：ASSESS 只做规模/工具/分区决策，不做内容级扫描；实际的基线扫描由 Dispatcher 按该策略执行并写入 findings.json。

**策略决策**：
- 文件数 ≤ 10：直接扫描
- 文件数 10-50：Grep 预过滤
- 文件数 > 50：分区并行扫描
- 预估上下文 > 100K tokens：启用摘要模式

详见：[workflows/assess.md](../workflows/assess.md)

## Phase 1: 需求分析 (analyze)

**目标**：理解用户需求，澄清歧义，输出 RFC §1-§2

**专家**：需求分析师 (`personas/core/analyst-core.md`)

**关键活动**：
1. 复用 Dispatcher 基线扫描（scan-of-record），验证证据覆盖（≥3 且覆盖入口/配置/依赖或测试/可观测）
2. 仅在发现“证据缺口”时触发补扫（而非重复全量扫描）
3. 提出选项式问题（禁止开放式）
   - 格式："发现 [X] 种方案：A. [选项] (参考: repo@rev:path#Lline), B. [选项]..."
4. 定义功能需求和验收标准
5. 识别非功能需求
6. 明确范围与排除项
7. 记录假设和约束
8. 提炼成功指标/业务收益（如适用）

**输出**：RFC §1（背景）+ §2（需求）

**质量标准**：
- ✅ 基线证据充分：key_files ≥3 且覆盖面达标；facts ≥3 且每条附证据锚点
- ✅ 所有问题都是选项式
- ✅ 需求具体可测量
- ✅ 验收标准可测试
- ✅ 验收标准为可观察结果（The What），非实现步骤/技术选型（The How）
- ✅ 范围/排除项明确

## Phase 2: 方案设计 (design)

**目标**：设计技术方案，定义接口，输出 RFC §3-§5

**专家**：架构设计师 (`personas/core/architect-core.md`)

**关键活动**：
1. 设计 API 契约（接口、参数、返回值）
2. 定义数据结构和状态管理
3. 指定错误处理和边界情况
4. 记录 DFX 考量（8维度）
5. 明确灰度/开关策略（如适用）
6. 创建设计决策及理由

**输出**：RFC §3（契约）+ §4（DFX）+ §5（设计决策）

**质量标准**：
- ✅ API 契约开发者友好
- ✅ DFX 需求量化（非模糊）
- ✅ 覆盖全部 8 个 DFX 维度
- ✅ 设计决策包含备选方案和权衡

## Phase 3: 自动预检 (precheck)

**目标**：验证 RFC 结构、场景覆盖、DFX 完整性

**执行者**：编排器（自动检查，无需子代理）

**关键活动**：
1. 检查 RFC 结构完整性（5个章节）
2. 验证场景覆盖（正常/异常/边界/不支持）
3. 验证 Gherkin 格式（#### Scenario: + GIVEN/WHEN/THEN）
4. 检查 DFX 维度完整性（8维度）
5. 检测模糊描述

**输出**：预检报告（通过/失败）

**阻塞条件**：任何检查失败则退回修改

详见 [precheck.md](../workflows/precheck.md)

---

## Phase 4: 质量审查 (audit)

**目标**：全面质量审查，评分，审批决策

**专家**：质量审查员 (`personas/core/auditor-core.md`)

**关键活动**：
1. 对照质量清单审查 RFC
2. 验证技术可行性和一致性
3. 检查歧义和缺口
4. 多维度评分（0-100）
5. 给出审批决策

**输出**：审查报告（评分+决策）

**决策标准**：
- ≥90分：通过
- <80分：退回修改
- 80-89分：需讨论（用户决定）

## Phase 5: 导出机读件 (export)

**目标**：在审查通过后，手动导出派生的机读件（供后续编码/验证 Agent 消费）。

**执行者**：编排器（手动触发）

**输出**：
- `rfc.digest.json`（稳定 schema，包含 REQ/SCN/API/CHG/EDGE/DEP/TST 等结构化事实与 provenance）
- `tasks.json`（可选，任务清单骨架）

**原则**：
- RFC (`rfc.md`) 仍是唯一事实来源；机读件为派生物，不在 RFC 内重复维护 YAML/JSON
- 默认不自动触发导出（遵循“仅手动 export”决策）

详见：[workflows/export.md](../workflows/export.md)
