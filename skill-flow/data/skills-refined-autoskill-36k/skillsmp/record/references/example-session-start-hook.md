# 例: SessionStartフックの導入記録

## Summary

SessionStartフックを追加し、セッション開始時に `SESSION_ID` と `TRANSCRIPT_PATH` を環境変数として設定する仕組みを導入した。

## 背景と動機

- Claude自身が自分のセッションIDと会話ログのパスを参照できるようにしたい
- 別のClaudeプロセスから現在のセッションの会話ログを読み取って要約させるなど、セッション間連携の基盤にしたい
- 親子セッション管理（親が子のsession_idで `--resume` する等）の基盤にしたい

## 調査と議論の経緯

### session_id と transcript_path の使い分け

- `session_id`: `claude --resume <id>` でセッションを継続するための識別子
- `transcript_path`: 会話ログ（JSONL）の実ファイルパス

別プロセスから会話内容を読む用途では `transcript_path` を直接渡すべきという結論に至った。session_idからファイルパスを導出する公式な方法がないため。

### 情報の公開先として何を使うか

3つの選択肢を検討した：

| 方法 | Claudeが認識するか | ユーザーに見えるか | 永続性 |
|------|-------------------|-------------------|--------|
| `additionalContext` | する | しない | コンパクト時に消える可能性あり |
| `CLAUDE_ENV_FILE`（環境変数） | しない（変数名を知らなければ） | しない | セッション中 |
| CLAUDE.md に記載 | する | する | 常時 |

### additionalContext を採用しなかった理由

- 環境変数に値が入っている以上、Claudeが変数名さえ知っていればいつでも `echo` で取得できる
- CLAUDE.md に変数名と用途を書いておけば、Claudeは常にその存在を認識する
- additionalContext はコンパクト時に消える可能性があり、CLAUDE.md の方が確実
- したがって additionalContext による二重管理は不要と判断した

### フックからユーザーに直接表示する方法について

- フックの仕組み上、ユーザーのターミナルに直接テキストを表示する機能は存在しない
- `type: "prompt"` を使えばClaudeに「セッションIDを伝えなさい」と指示することは可能だが、確実に従う保証がないため採用しなかった
- 必要な場合はユーザーが「セッションIDを教えて」と聞けばClaudeがCLAUDE.mdから変数名を読み取って取得する運用とした

### matcher フィールドについて

公式ドキュメントにより、SessionStart はマッチャー不要のイベントであることを確認した。PreToolUse 等のツール系イベントとは異なり、matcher フィールドを省略する。
