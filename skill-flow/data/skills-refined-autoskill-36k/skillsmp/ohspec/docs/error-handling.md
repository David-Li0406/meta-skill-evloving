# 错误处理协议

## 错误分类

| 级别 | 类型 | 示例 | 处理策略 |
|------|------|------|----------|
| L1 | 可自动恢复 | 工具调用超时、网络抖动、临时文件锁 | 自动重试（最多3次，指数退避） |
| L2 | 需降级处理 | 代码扫描结果不足、上下文超限、部分工具不可用 | 降级到简化模式继续执行 |
| L3 | 需用户介入 | 需求歧义、多方案需决策、关键信息缺失 | 暂停并请求澄清 |
| L4 | 不可恢复 | 权限不足、目标文件不存在、依赖服务宕机 | 终止并报告 |

## 3-Strike 规则

连续 3 次相同错误后，必须升级处理策略：

```
L1 错误 ×3 → 升级为 L2（尝试降级处理）
L2 错误 ×3 → 升级为 L3（请求用户介入）
L3 错误 ×3 → 升级为 L4（终止任务）
```

## 各级别处理流程

### L1 可自动恢复

```
检测到错误 → 记录日志 → 等待(2^n秒) → 重试 → 成功则继续/失败则计数+1
```

重试策略：
- 最大重试次数：3
- 退避间隔：2s → 4s → 8s
- 重试前检查：确认错误条件是否仍存在

### L2 需降级处理

```
检测到错误 → 记录日志 → 评估降级方案 → 执行降级 → 继续执行（标记降级状态）
```

降级策略表：

| 原始操作 | 降级方案 |
|----------|----------|
| 完整代码扫描 | 仅扫描关键路径 |
| 多专家协作 | 单专家简化流程 |
| 详细分析 | 快速摘要模式 |
| 完整 RFC | 精简 RFC |
| Task 子代理不可用 | 编排器最小化扫描回退（rg/ag/grep），先输出候选锚点/模块供确认，记录 audit_log/tooling |

### L3 需用户介入

```
检测到错误 → 记录日志 → 生成澄清请求 → 暂停执行 → 等待用户响应 → 继续/调整
```

澄清请求格式：
```markdown
## 需要澄清

**问题**: [具体问题描述]
**影响**: [对任务的影响]
**选项**:
1. [选项A]
2. [选项B]
**建议**: [推荐选项及理由]
```

### L4 不可恢复

```
检测到错误 → 记录日志 → 保存当前状态 → 生成错误报告 → 终止任务
```

## 错误日志格式

```yaml
timestamp: "2024-01-16T10:30:00Z"
level: L1|L2|L3|L4
phase: dispatcher|analysis|design|review
error_type: timeout|insufficient_data|ambiguity|permission_denied
message: "错误描述"
context:
  task_id: "xxx"
  agent: "dispatcher|analyst|architect|reviewer"
  operation: "具体操作"
action_taken: "采取的措施"
result: success|escalated|terminated
strike_count: 1|2|3
```

## 阶段特定错误处理

### Dispatcher 阶段
- 代码扫描超时 → L1 重试
- 扫描结果为空 → L2 扩大搜索范围
- 无法确定复杂度 → L3 请求用户提供更多信息

### 分析阶段
- 需求理解歧义 → L3 请求澄清
- 上下文不足 → L2 降级为快速分析

### 设计阶段
- 多方案冲突 → L3 请求用户决策
- 技术约束冲突 → L2 记录约束，继续最佳努力方案

### 审查阶段
- 质量门禁未通过 → 返回上一阶段修正（非错误，正常流程）
- 审查工具不可用 → L2 降级为人工审查清单
