# 执行计划

## 概述

加载计划，批判性审查，分批执行任务，并在批次之间报告以供审查。

**核心原则：** 带检查点的批量执行，供主笔（架构师）审阅。

**开始时声明：** “我正在使用 executing-plans 技能来实施此计划。”

## 流程

### 步骤 1: 加载和审阅计划

#### 1.1 格式验证（必须通过）

计划文件**必须**符合 writing-plans 格式，检查以下必要元素：

| 必要元素 | 验证方式 |
|----------|----------|
| 标题区块 | 有 `**目标：**`、`**架构：**`、`**技术栈：**` |
| 任务结构 | 有 `### 任务 N：[名称] 🟢/🟡/🔴` 格式 |
| 文件清单 | 每个任务有 `**文件：**` 区块 |
| 步骤明细 | 每个任务有 `**步骤 N：**` 格式 |
| 风险标注 | 每个任务有 🟢/🟡/🔴 标记 |
| 变更摘要 | 结尾有 `## 变更摘要` 区块 |

**如果格式不符：**
- 停止执行
- 告知用户：“此计划不符合 writing-plans 格式，缺少：[列出缺少元素]”
- 建议：“请使用 writing-plans 技能重新生成计划，或补齐缺少的元素。”

#### 1.2 内容审查

格式验证通过后：
1. 阅读计划文件
2. 批判性审查 - 识别对计划的任何问题或疑虑
3. 如果有疑虑：在开始之前向你的人类伙伴提出
4. 如果无疑虑：创建 TodoWrite 并继续

### 步骤 2: 执行批次
**默认：前 3 个任务**

对于每个任务：
1. 标记为 in_progress
2. 严格遵循每个步骤（计划有详细的步骤）
3. 运行指定的验证
4. 标记为 completed

### 风险等级执行控管

**🔴 高风险任务：**
1. 暂停并向用户单独确认：“任务 N 是高风险变更，涉及 [具体影响]。确认执行？”
2. 确认备份/回滚指令已就绪
3. 获得明确同意后才执行
4. 执行后立即验证，报告结果并等待确认再继续

**🟡 中风险任务：**
1. 提醒：“执行任务 N（中风险）：[简述影响]”
2. 确认相关测试存在
3. 执行后在批次报告中特别标注结果

**🟢 低风险任务：**
- 可直接执行，无需额外确认
- 完成后纳入批次报告

### 步骤 3: 报告
当批次完成时：
- 展示已实施的内容
- 展示验证输出
- 特别标注 🟡/🔴 任务的执行结果
- 说：“准备好接受反馈。”

### 步骤 4: 继续
根据反馈：
1. 如果需要，应用更改
2. 执行下一个批次
3. 重复直到完成

### 步骤 5: 完成开发

所有任务完成并验证后：
1. **检查分支状态**：
   ```bash
   git branch --show-current
   ```

2. **根据分支决定下一步**：

   **如果在功能分支（非 main/master）：**
   - 宣布：“我正在使用 finishing-a-development-branch 技能来完成这项工作。”
   - **必要子技能：** 使用 hi-skills:finishing-a-development-branch
   - 遵循该技能以验证测试、呈现选项、执行选择

   **如果在主分支（main/master）：**
   - 报告完成摘要：已完成的任务、提交记录
   - 询问用户：“变更已在本地 main 分支。要推送到远端吗？”
   - 根据用户选择执行 `git push` 或保持现状

## 何时停止并寻求帮助

**当出现以下情况时立即停止执行：**
- 批次中间遇到阻碍（缺少依赖项、测试失败、指令不清楚）
- 计划有关键缺口导致无法开始
- 你不理解某个指令
- 验证重复失败

**寻求澄清而不是猜测。**

## 何时重新访问早期步骤

**在以下情况返回审阅（步骤 1）：**
- 合作伙伴根据你的反馈更新了计划
- 基本论证方法需要重新思考

**不要强行突破阻碍** - 停止并询问。

## 记住
- 先批判性地审阅计划
- 严格遵循计划步骤
- 不要跳过验证
- 当计划要求时引用技能
- 批次之间：只报告并等待
- 受阻时停止，不要猜测
