## RFC 结构要求

所有 RFC 必须包含 5 个章节：

| 章节 | 内容 |
|------|------|
| §1 背景与动机 | 背景、动机、范围、排除项 |
| §2 需求定义 | 功能需求、非功能需求、假设、依赖 |
| §3 接口契约 | 契约表、对齐与变更矩阵、场景规格、错误码 |
| §4 DFX | 8维度分析、量化需求、验证方法 |
| §5 设计决策 | 备选方案、选定方案及理由、权衡、风险 |

## DFX 8 维度

| 维度 | 检查要点 |
|------|----------|
| 安全性 | 认证、授权、加密、数据保护 |
| 可靠性 | 错误处理、容错、恢复机制 |
| 性能 | 延迟、吞吐量、资源使用、可扩展性 |
| 可维护性 | 代码清晰度、模块化、文档 |
| 可测试性 | 单元测试、集成测试、覆盖率 |
| 可观测性 | 日志、监控、追踪、指标 |
| 兼容性 | 向后兼容、版本迁移、废弃策略 |
| 可运维性 | 灰度/开关/回滚/运维手册 |

**量化要求**（禁止模糊描述）：
- ❌ "性能良好" → ✅ "p99延迟 < 10ms"
- ❌ "需要权限" → ✅ "`ohos.permission.MANAGE_AUDIO`"

## 场景覆盖要求

必须包含 4 种场景类型（Gherkin 格式）：

1. **正常场景**：预期使用路径
2. **异常场景**：错误条件、权限拒绝、无效输入
3. **边界场景**：边界情况、并发访问、资源限制
4. **不支持场景**：硬件限制、功能不可用

**Gherkin 格式**：
```markdown
#### Scenario: [场景名称]
- **GIVEN** [前置条件]
- **WHEN** [动作]
- **THEN** [预期结果]
- **REASON** [不支持原因，如适用]
```

## 强制检查清单

审批前必须验证：

- [ ] 5 个 RFC 章节完整且非空
- [ ] 8 个 DFX 维度有量化需求
- [ ] 至少 4 种场景类型覆盖
- [ ] 所有场景遵循 Gherkin 格式
- [ ] 无模糊描述
- [ ] 范围/排除项明确
- [ ] 每条功能需求都有验收标准
- [ ] 验收标准为可观察结果（The What），不得写实现步骤/技术选型（The How）
- [ ] 现有模式证据：至少 1 个相似实现/现有模式锚点；若不存在需声明 no-prior-art + 搜索范围/关键词
- [ ] 所有问题都是选项式
- [ ] API 契约开发者友好
- [ ] 开发者模型与易用性说明（目标开发者/心智模型/误用预防/错误信息）
- [ ] 设计决策包含备选方案和理由

## 补充建议（非阻断）
- [ ] 审查速览包含成功指标/业务收益
- [ ] 如涉及跨团队/陌生术语，补充术语/缩写表
- [ ] 可运维性明确灰度/开关策略（如适用）

**新增强制门禁（工业级落地）**：
- [ ] **可执行门禁**：必须通过 `.ohspec/scripts/precheck_rfc.py`（无 error）；COMPLEX 建议 `--strict`（兼容：`.claude/ohspec/scripts/precheck_rfc.py`）
- [ ] **证据覆盖面**：key_files ≥ 3 且覆盖入口/配置/依赖（或测试/可观测）
- [ ] **图示要求**：COMPLEX 必须提供 Mermaid 图示（交互流程或状态机）
- [ ] **审查速览**：包含场景/规格/开发者模型的快速索引与缺口提示
- [ ] **场景概览**：需求层提供场景标题清单，便于快速审查覆盖面
- [ ] **项目事实**：提供 FACT-ID 事实清单，每条附证据锚点（证明理解现状，避免“按模板想象”）
- [ ] **需求化输出（少代码）**：RFC 以契约/矩阵为中心；除 Mermaid 外禁止大量实现代码（否则视为不可审查/不可追溯）
- [ ] **ID 规范**：REQ/SCN/API/CHG/EDGE/DEP/TST/FIELD 使用稳定 ID；追溯矩阵引用必须存在，任何断链视为缺口
- [ ] **对齐与变更矩阵**：每条变更必须提供 existing_ref 或 new_change_plan
- [ ] **依赖影响矩阵**：外部依赖的失败模式、回退策略、观测与影响范围齐全
- [ ] **异常/边界矩阵**：并发/时序/重连/权限/多用户等边界覆盖
- [ ] **测试矩阵**：场景→用例→验收指标闭环，包含依赖故障注入与兼容回归
- [ ] **追溯矩阵**：REQ/SCN/TST 可追溯，任何断链视为缺口
- [ ] **多仓一致性（如适用）**：Repo Manifest 完整；锚点统一为 repo@rev:path#Lline；跨仓影响与发布顺序明确

**阻塞**：任何检查失败则退回修改。
