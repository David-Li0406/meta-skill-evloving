**版本**：v3.0

# 第一部分：快速开始

## 🚀 两种使用模式（选择你的方式）

### 模式 1：YOLO 模式（推荐，一键执行）

**适用场景**：
- 需求相对明确
- 信任 AI 的判断
- 希望快速获得结果

**使用方法**：
```bash
/ohspec "为音频服务增加 3D 音效开关"
```

**执行流程**：
```
用户输入需求
  ↓
自动扫描代码库
  ↓
【可能澄清】如果需求模糊，会提出 1-3 个选项式问题
  ↓
自动生成完整 RFC（§1-§5）
  ↓
输出结果
```

可选：当 RFC 审查通过后，可手动执行 `/ohspec:export` 导出 `rfc.digest.json`（以及可选的 `tasks.json`），用于后续编码 Agent 消费。

**澄清机制**：
- ✅ **会澄清**：如果需求模糊或有多种实现方式
- ✅ **简化澄清**：只问关键问题（1-3 个），不会问太多
- ✅ **选项式**：提供 2-4 个选项，不是开放式问题
- ⏭️ **可能跳过**：如果需求足够明确（触发 SIMPLIFY_CLARIFY 信号）

**时间**：
- 简单任务：5-8 分钟
- 中等任务：15-20 分钟
- 复杂任务：1-2 小时

---

### 模式 2：手动模式（精细控制）

**适用场景**：
- 需求复杂，需要分阶段确认
- 希望在每个阶段审查结果
- 需要精细控制流程

**使用方法**：
```bash
# 阶段 1：需求分析
/ohspec:analyze "需求描述"
# → 审查 RFC §1-§2，确认无误后继续

# 阶段 2：方案设计
/ohspec:design RFC-20260119-143000
# → 审查 RFC §3-§5，确认无误后继续

# 阶段 3：质量门禁
/ohspec:audit RFC-20260119-143000
# → 审查审查报告，修复问题

# （可选）阶段 4：导出派生机读件（不自动触发）
/ohspec:export RFC-20260119-143000
/ohspec:export RFC-20260119-143000 --tasks
```

**时间**：
- 比 YOLO 模式慢 20-30%
- 但控制更精细

---

## 🔄 修改路径（如果不满意）

### 场景 1：需求理解有误（RFC §1-§2 不对）

**修改路径**：
```bash
# 方法 1：重新执行 analyze 阶段（推荐）
/ohspec:analyze "需求描述（补充更多细节）"

# 方法 2：手动编辑 RFC
# 编辑 RFC-xxx.md 的 §1-§2
# 然后继续执行 design 阶段
/ohspec:design RFC-xxx
```

### 场景 2：设计方案不满意（RFC §3-§5 不对）

**修改路径**：
```bash
# 方法 1：重新执行 design 阶段（推荐）
/ohspec:design RFC-xxx

# 方法 2：手动编辑 RFC
# 编辑 RFC-xxx.md 的 §3-§5
# 然后执行 audit 验证
/ohspec:audit RFC-xxx
```

### 场景 3：质量门禁不通过

**修改路径**：
```bash
# 方法 1：根据 audit 报告修复问题
# 手动编辑 RFC-xxx.md，修复问题
/ohspec:audit RFC-xxx

# 方法 2：重新执行 design 阶段
/ohspec:design RFC-xxx
```

### 场景 4：完全不满意（推倒重来）

**修改路径**：
```bash
# 重新执行 YOLO 模式，提供更详细的需求
/ohspec "需求描述（补充更多细节、约束条件、示例）"

# 或使用手动模式，分阶段确认
/ohspec:analyze "需求描述"
```

## 📊 使用模式对比

| 维度 | YOLO 模式 | 手动模式 |
|------|-----------|----------|
| **速度** | 快（5-20 分钟） | 慢（+20-30%） |
| **控制** | 低（自动执行） | 高（分阶段确认） |
| **澄清** | 简化（1-3 个问题） | 完整（所有必要问题） |
| **适用** | 简单/中等任务 | 复杂任务 |
| **修改** | 重新执行或手动编辑 | 分阶段重新执行 |
| **推荐** | ✅ 大多数场景 | 复杂/关键项目 |

## 💡 最佳实践

### 1. 需求描述要清晰

**好的需求描述**：
```
为音频服务增加 3D 音效开关
- 支持全局和应用级作用域
- 参考 Android AudioManager 的设计
- 需要考虑性能影响（延迟 < 50ms）
```

**不好的需求描述**：
```
增加音效功能
```

### 2. 先用 YOLO，不满意再手动

- 大多数情况下，YOLO 模式足够好
- 如果不满意，根据修改路径调整
- 不要一开始就用手动模式（除非需求非常复杂）

### 3. 善用修改路径

- 不要害怕修改，RFC 是可以迭代的
- 小问题手动编辑，大问题重新执行
- 每次修改后都执行 audit 验证

### 4. 操作系统类项目（竞品分析）

**自动触发**：
- 如果需求包含"API"、"接口"、"性能"、"架构"、"安全"等关键词
- 会自动加载 Android/iOS/Windows 的设计模式
- 提供 3-5 条关键要点

**示例**：
```bash
/ohspec "设计音频焦点管理 API，参考 Android/iOS 最佳实践"
# → 自动触发竞品分析
# → 加载 knowledge-base/android/api-design.md
# → 加载 knowledge-base/ios-macos/api-design.md
# → 提供关键要点
```

## 🆘 快速帮助

**问题 1**：YOLO 模式下，AI 没有问我任何问题就生成了 RFC，但结果不对
- **原因**：需求被判断为足够明确（触发 SIMPLIFY_CLARIFY）
- **解决**：重新执行，提供更详细的需求描述，或使用手动模式

**问题 2**：我想修改 RFC 的某个部分，但不想重新执行整个流程
- **原因**：只需要局部修改
- **解决**：手动编辑 RFC-xxx.md，然后执行 /ohspec:audit RFC-xxx 验证

**问题 3**：audit 报告显示有很多问题，我该怎么办？
- **原因**：RFC 质量不达标
- **解决**：根据 audit 报告逐个修复，或重新执行 design 阶段

**问题 4**：我不知道该用 YOLO 还是手动模式
- **原因**：不确定任务复杂度
- **解决**：先用 YOLO，如果不满意再用手动模式

# 第二部分：详细参考


## 完整使用流程

### 阶段 1：需求分析（analyze）

**目标**：理解需求，输出 RFC §1-§2

**步骤**：
1. 系统自动扫描代码库（使用 Grep/Glob）
2. 识别相关文件和现有实现（至少 3 个且覆盖入口/配置/依赖或测试/可观测）
3. 【可能触发竞品分析】如果需求包含 API/架构/性能/安全关键词
4. 基于扫描结果提供选项式问题（如有必要）
5. 用户选择选项，澄清需求
6. 输出 RFC §1（上下文）和 §2（需求）

**输出物**：
- `rfc.md` - RFC §1-§2
- `findings.json` - 代码扫描发现（可能包含 competitive_analysis）
- `progress.json` - 分析过程记录

**质量标准**：
- ✅ findings.json 包含至少 3 个相关文件且覆盖面达标
- ✅ 所有提问都是选项式（2-4 个选项）
- ✅ 每个选项都引用代码示例
- ✅ 需求无歧义

---

### 阶段 2：方案设计（design）

**目标**：设计契约和接口，输出 RFC §3-§5

**步骤**：
1. 基于 §1-§2 设计接口和数据结构
2. 定义场景规格（Gherkin 格式）
3. 补充 DFX 约束（8 个维度）
4. 记录设计决策（ADR）
5. 输出 RFC §3-§5

**输出物**：
- `rfc.md` - 完整 RFC（§1-§5）

**质量标准**：
- ✅ 接口定义清晰（参数、返回值、错误码）
- ✅ 场景覆盖完整（正常/异常/边界/不支持）
- ✅ DFX 约束量化（无模糊描述）
- ✅ 设计决策有理由

---

### 阶段 3：质量门禁（audit）

**目标**：检查 RFC 质量，确保符合标准

**步骤**：
1. 结构完整性检查（5 个章节）
2. 场景覆盖检查（4 类场景）
3. DFX 清单检查（8 个维度）
4. 模糊描述检测（自动检测）
5. 生成审查报告

**输出物**：
- `audit-report.md` - 审查报告

**质量标准**：
- ✅ 无阻塞问题（error 级别）
- ✅ 警告问题已修复或说明
- ✅ 所有 DFX 约束量化

---

### 阶段 4：预检（precheck）

**目标**：自动检查 RFC 质量，提前发现问题

**步骤**：
1. 自动检查结构完整性
2. 自动检查场景覆盖
3. 自动检查 DFX 清单
4. 自动检测模糊描述
5. 生成预检报告

**输出物**：
- `precheck-report.md` - 预检报告

**质量标准**：
- ✅ 无阻塞问题
- ✅ 所有检查通过

## 故障排查

### 问题 1：findings.json 不足 3 个文件

**错误信息**：
```
❌ findings.json 仅包含 1 个文件引用，至少需要 3 个。
```

**原因**：
- 代码扫描范围太窄
- 关键词不准确

**解决方案**：
1. 扩大搜索范围（使用更通用的关键词）
2. 使用 Glob 查找相关文件
3. 阅读更多相似实现

**示例**：
```bash
# 扩大搜索范围
Grep pattern="audio|sound|3d" output_mode="files_with_matches"
Glob pattern="**/*audio*.{cpp,h}"
```

---

### 问题 2：提问不符合模板格式

**错误信息**：
```
❌ 提问缺少选项（至少需要 2 个选项 A/B）
```

**原因**：
- 使用了开放式提问
- 未引用代码示例

**解决方案**：
1. 将开放式问题改为选项式
2. 为每个选项引用代码示例
3. 说明每个选项的优缺点

**示例**：
```
❌ 错误：你希望如何实现？

✅ 正确：
根据代码扫描，发现 2 种实现方式：
A. 同步处理（参考：AudioService.cpp:123）
   - 优点：延迟低
   - 缺点：阻塞主线程
B. 异步队列（参考：AudioQueue.cpp:456）
   - 优点：非阻塞
   - 缺点：增加复杂度

你希望使用哪种方式？
```

---

### 问题 3：DFX 约束被标记为模糊

**错误信息**：
```
❌ 性能描述模糊："延迟低"
```

**原因**：
- 未量化性能指标
- 缺少具体数值

**解决方案**：
1. 提供具体数值（< 100ms）
2. 说明百分位（p99 < 50ms）
3. 提供资源预算（内存 < 10MB）

**示例**：
```
❌ 错误：延迟低

✅ 正确：
- p50 延迟 < 20ms
- p99 延迟 < 50ms
- p99.9 延迟 < 100ms
```

---

### 问题 4：场景覆盖不全

**错误信息**：
```
❌ 缺少异常场景（必须）
```

**原因**：
- 仅考虑了正常流程
- 未考虑错误处理

**解决方案**：
1. 补充异常场景（权限不足、参数错误等）
2. 补充边界场景（并发、极限值等）
3. 补充不支持场景（如适用）

**示例**：
```markdown
#### Scenario: 权限不足时启用 3D 音效
- **GIVEN** 用户无 MANAGE_AUDIO 权限
- **WHEN** 调用 enable3DSound()
- **THEN** 返回错误码 401，提示"权限不足"
```

---

### 问题 5：RFC 格式错误

**错误信息**：
```
❌ 场景标题必须使用 "#### Scenario:" 格式
```

**原因**：
- 未使用 Gherkin 格式
- 标题格式不正确

**解决方案**：
1. 使用 `#### Scenario:` 作为标题
2. 使用 GIVEN/WHEN/THEN 结构
3. 参考模板文件

**示例**：
```markdown
❌ 错误：
### 场景 1：启用 3D 音效
前置条件：用户有权限
操作：调用接口
结果：成功

✅ 正确：
#### Scenario: 启用 3D 音效
- **GIVEN** 用户有 MANAGE_AUDIO 权限
- **WHEN** 调用 enable3DSound()
- **THEN** 3D 音效启用，返回成功
```

## 高级用法

### 1. 分阶段执行

如果需要更细粒度的控制，可以分阶段执行：

```bash
# 阶段 1：需求分析
/ohspec:analyze "需求描述"

# 阶段 2：方案设计
/ohspec:design RFC-20260114-143000

# 阶段 3：质量门禁
/ohspec:audit RFC-20260114-143000

# 阶段 4：预检
/ohspec:precheck RFC-20260114-143000
```

### 2. Spike 验证

如果技术不确定性高，可以先进行 Spike 验证：

```bash
/ohspec:spike "验证 Dolby Atmos SDK 集成可行性"
```

### 3. 自定义专家组合

调度员会根据需求自动选择专家，但你也可以手动指定：

```bash
/ohspec:start "需求描述" --experts="analyst,architect,api_designer"
```

### 4. 性能优化模式（Resume Mode）

对于简单和中等任务，系统会自动使用 Resume 模式优化性能：

**工作原理**：
- 单个 subagent 连续执行多个阶段（通过 `resume` 参数）
- 避免重复加载 persona 上下文
- 节省约 70% 的上下文加载开销（约 7,400 tokens）

**自动触发条件**：
- SIMPLE 任务：analyze + design 合并执行
- MEDIUM 任务：自动流转的阶段（无需用户确认）
- COMPLEX 任务：不使用 Resume（需要每阶段用户确认）

**性能提升**：
- 简单任务：执行速度提升 2-3 倍
- 中等任务：执行速度提升 1.5-2 倍
- 上下文消耗：从 ~11,200 tokens 降至 ~3,800 tokens

**用户无需操作**：调度员会根据复杂度自动选择最优模式，并默认自动流转到 precheck → audit（如无未决问题）。

**技术细节**：
如果你对实现感兴趣，可以查看 SKILL.md 中的"Resume 模式使用指南"章节（约 237 行详细说明）。

## 输出物说明

### 1. rfc.md

完整的 RFC 文档，包含 5 个章节（需求化设计）：
- §1 上下文
- §2 需求（含依赖与影响）
- §3 契约（契约表/对齐矩阵/场景规格）
- §4 DFX 约束（含测试矩阵）
- §5 设计决策

### 2. findings.json

代码扫描发现，包含：
- 相关文件列表（至少 3 个且覆盖面达标）
- 现有实现模式
- 依赖关系分析
- 关键代码片段
- 竞品分析（如触发）

### 3. progress.json

分析过程记录，包含：
- 阶段进度和状态（pending/in_progress/complete）
- 专家角色和执行时间
- Subagent ID（用于追踪和 Resume）
- Resume 模式标记（性能优化）
- 提问和回答
- 决策记录
- 风险识别

### 4. audit-report.md

审查报告，包含：
- 通过项
- 警告项
- 阻塞项
- 改进建议

## 常见问题（FAQ）

### Q1：ohspec 适合什么场景？

**A**：适合需要详细设计文档的场景：
- 新功能开发
- API 设计
- 架构改造
- 跨团队协作
- 操作系统类项目（支持竞品分析）

### Q2：生成 RFC 需要多长时间？

**A**：取决于复杂度（已优化，使用 Resume 模式）：
- 简单功能：5-8 分钟（优化前：10 分钟）
- 中等功能：15-20 分钟（优化前：30 分钟）
- 复杂功能：1-2 小时（需要每阶段用户确认，不使用 Resume）

### Q3：可以修改生成的 RFC 吗？

**A**：可以，RFC 是 Markdown 文件，可以手动编辑。建议：
- 保留结构和格式
- 补充细节
- 修正错误

### Q4：如何确保 RFC 质量？

**A**：ohspec 提供多层质量保障：
- 自动代码扫描（强制执行）
- 选项式提问（避免歧义）
- 模糊描述检测（自动检测）
- 质量门禁（audit 阶段）
- 预检（precheck 阶段）

### Q5：可以集成到 CI/CD 吗？

**A**：可以，使用 precheck 命令：
```bash
/ohspec:precheck RFC-20260114-143000
```
如果有阻塞问题，返回非零退出码。

### Q6：竞品分析什么时候触发？

**A**：自动触发条件：
- 需求包含关键词：API、接口、性能、架构、安全、设计模式
- 适用于操作系统类项目
- 在 analyze 阶段的 subagent 中执行
- 如果上下文使用量 > 70%，会跳过以避免爆上下文

### Q7：如果用户给的需求很详细，但对异常场景没有描述，会澄清还是自行决策？

**A**：会澄清，但方式取决于模式：
- **YOLO 模式**：会提出 1-2 个关键问题（简化澄清）
  - 例如："发现需求未提及异常处理，建议补充：A. 权限不足时返回错误码 B. 参数错误时抛出异常"
- **手动模式**：会提出完整的澄清问题
  - 例如：列出所有可能的异常场景，让用户选择需要处理哪些
- **自行决策**：如果需求足够详细且符合最佳实践，可能会自行补充常见异常场景（如权限不足、参数错误），但会在 RFC 中标注"推断"

**建议**：
- 如果希望 AI 自行决策，在需求中明确说明"按照最佳实践补充异常场景"
- 如果希望精细控制，使用手动模式并仔细审查每个阶段
