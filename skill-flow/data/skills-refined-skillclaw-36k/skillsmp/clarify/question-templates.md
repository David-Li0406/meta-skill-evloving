## 定義の明確化

**目的**: 曖昧な言葉を具体的な定義に落とし込む

### テンプレート

#### 基本形

```markdown
### 質問: 「[用語]」の定義

「[用語]」とは具体的に何を指しますか？

**考慮すべき観点:**

- [観点1]
- [観点2]

**例:**

- [具体例A]: [説明]
- [具体例B]: [説明]
```

#### 適用例

```markdown
### 質問: 「好み」の定義

「好み」とは具体的に何を指しますか？

**考慮すべき観点:**

- 明示的な意思表示（Like/Dislike）
- 行動から推測される嗜好（閲覧時間、再訪問）
- 過去の選択履歴

**例:**

- 明示的好み: ユーザーがLikeボタンを押した記事
- 暗黙的好み: 3分以上閲覧した記事
```

```markdown
### 質問: 「成功」の定義

この機能における「成功」とは何で判断しますか？

**考慮すべき観点:**

- ユーザー満足度
- ビジネス指標
- 技術的パフォーマンス

**例:**

- ユーザー視点: 欲しい情報がすぐ見つかる
- ビジネス視点: コンバージョン率向上
- 技術視点: レスポンス時間 < 500ms
```

## シグナルの優先順位

**目的**: 複数の入力シグナルの優先度を決定する

### テンプレート

```markdown
### 質問: シグナルの優先順位

以下のシグナルをどのような優先順位で扱いますか？

| シグナル | 例   | 信頼度 |
| -------- | ---- | ------ |
| 明示的   | [例] | 高     |
| 暗黙的   | [例] | 中     |
| 推測     | [例] | 低     |

**選択肢:**

- A: 明示的シグナルのみ使用
- B: 明示的を優先、暗黙的を補助的に使用
- C: 両方を同等に扱う
```

#### 適用例

```markdown
### 質問: ユーザー嗜好シグナルの優先順位

ユーザーの「好み」を判断する際、以下のシグナルをどう優先しますか？

| シグナル | 例                                 | 信頼度 |
| -------- | ---------------------------------- | ------ |
| 明示的   | Like/Dislikeボタン、お気に入り登録 | 高     |
| 暗黙的   | 閲覧時間、スクロール深度           | 中     |
| 推測     | 類似ユーザーの行動                 | 低     |

**選択肢:**

- A: 明示的シグナルのみ使用（精度重視）
- B: 明示的を優先、暗黙的を補助的に使用（バランス）
- C: 両方を同等に扱う（データ量重視）
```

## コールドスタート

**目的**: データ不足時の対応戦略を決定する

### テンプレート

```markdown
### 質問: コールドスタート戦略

[対象]のデータがない状態で、どう対応しますか？

**選択肢:**

- A: [戦略A] - [メリット/デメリット]
- B: [戦略B] - [メリット/デメリット]
- C: [戦略C] - [メリット/デメリット]

**考慮事項:**

- 初期体験の品質
- データ収集の速度
- 実装コスト
```

#### 適用例

```markdown
### 質問: 新規ユーザーへの初期推薦

新規ユーザー（行動履歴なし）への初期推薦はどうしますか？

**選択肢:**

- A: 人気順で表示 - 安全だが個別化なし
- B: ランダム表示 - 多様性あるがノイズも多い
- C: オンボーディングで好みを聞く - 精度高いが離脱リスク

**考慮事項:**

- 初期体験の品質: どれだけ早く価値を感じてもらえるか
- データ収集の速度: 個別化に必要なデータをどう集めるか
- 実装コスト: オンボーディングUIの開発工数
```

## 成功指標

**目的**: 測定可能なKPIを定義する

### テンプレート

```markdown
### 質問: 成功の測定方法

この機能の「成功」を何で測定しますか？

**指標候補:**
| 指標 | 計算方法 | 目標値 |
|------|----------|--------|
| [指標1] | [計算式] | [目標] |
| [指標2] | [計算式] | [目標] |

**選択肢:**

- A: [主要指標]を最優先
- B: [複数指標]のバランス
- C: 段階的に指標を追加
```

#### 適用例

```markdown
### 質問: 推薦品質の測定方法

「適切な推薦」を何で測定しますか？

**指標候補:**
| 指標 | 計算方法 | 目標値 |
|------|----------|--------|
| CTR | クリック数 / 表示数 | > 5% |
| 滞在時間 | 推薦経由の平均閲覧時間 | > 2分 |
| 再訪率 | 推薦経由で戻ってくる率 | > 30% |

**選択肢:**

- A: CTRを最優先（短期的効果重視）
- B: 滞在時間を重視（エンゲージメント重視）
- C: 複合指標でバランスを取る
```

## 時間的変化

**目的**: データの鮮度や減衰の扱いを決定する

### テンプレート

```markdown
### 質問: 時間経過による[対象]の扱い

[対象]は時間経過でどう変化させますか？

**考慮事項:**

- 減衰: どのくらいで古くなる？
- 復活: 再度有効になる条件は？
- 更新: いつ再計算する？

**選択肢:**

- A: 時間減衰あり（[期間]で半減）
- B: 時間減衰なし（永続）
- C: 明示的なリセットまで有効
```

#### 適用例

```markdown
### 質問: Dislikeの時間的扱い

ユーザーがDislikeした記事は、どのくらいの期間表示を抑制しますか？

**考慮事項:**

- 減衰: 永久にDislikeを維持？それとも時間で薄まる？
- 復活: 「もう一度見たい」と思う可能性は？
- 類似: Dislikeした記事に「類似」した記事の扱いは？

**選択肢:**

- A: 永久にDislike維持
- B: 6ヶ月で減衰（再表示される可能性あり）
- C: 明示的に「解除」するまで維持
```

## UIトレードオフ

**目的**: ユーザー負担と情報取得のバランスを決定する

### テンプレート

```markdown
### 質問: [機能]の主体

[機能]は誰が決定しますか？

**選択肢:**
| 主体 | メリット | デメリット |
|------|----------|------------|
| システム | [メリット] | [デメリット] |
| ユーザー | [メリット] | [デメリット] |
| ハイブリッド | [メリット] | [デメリット] |

**関連質問:**

- ユーザーが介入できる範囲は？
- デフォルト値は何が適切？
```

#### 適用例

```markdown
### 質問: 推薦比率の決定主体

「探索（新しいジャンル）vs 活用（好みのジャンル）」の比率は誰が決めますか？

**選択肢:**
| 主体 | メリット | デメリット |
|------|----------|------------|
| システム | ユーザー負担なし | 個人差に対応しづらい |
| ユーザー | 細かく調整可能 | 設定が面倒 |
| ハイブリッド | バランス良い | 実装が複雑 |

**関連質問:**

- ユーザーが変更できるとしたら、どの程度細かく？
- デフォルトの探索:活用比率は？
```

## 属性の考慮

**目的**: 副次的な属性の扱いを決定する

### テンプレート

```markdown
### 質問: [属性]の考慮

[主要属性]以外に、[副次属性]も考慮しますか？

**影響範囲:**

- [影響1]
- [影響2]

**選択肢:**

- A: 考慮しない（シンプル）
- B: フィルタとして使用（ユーザー選択）
- C: スコアリングに組み込む（自動調整）
```

#### 適用例

```markdown
### 質問: 記事の「長さ」の考慮

記事の長さ（読了時間）を推薦に考慮しますか？

**影響範囲:**

- 短い記事が好きなユーザーへの配慮
- 「今時間がある/ない」という状況への対応
- 長文コンテンツの露出機会

**選択肢:**

- A: 考慮しない（ジャンル・評価のみで推薦）
- B: フィルタとして使用（「10分以内」などユーザーが選択）
- C: 過去の閲覧パターンから自動推測
```

## UIの粒度

**目的**: ユーザーが設定できる細かさを決定する

### テンプレート

```markdown
### 質問: [設定]の粒度

ユーザーが[設定]を調整できる場合、どの程度細かくしますか？

**粒度レベル:**
| レベル | 例 | ユーザー負担 |
|--------|-----|--------------|
| 粗い | [例] | 低 |
| 中程度 | [例] | 中 |
| 細かい | [例] | 高 |

**考慮事項:**

- 設定に費やす時間
- 設定を変更する頻度
- パワーユーザー vs 一般ユーザー
```

#### 適用例

```markdown
### 質問: ジャンル好みの設定粒度

ユーザーがジャンルの好みを設定する場合、どの程度細かくしますか？

**粒度レベル:**
| レベル | 例 | ユーザー負担 |
|--------|-----|--------------|
| 粗い | 好き/嫌い の2択 | 低 |
| 中程度 | 5段階評価 | 中 |
| 細かい | スライダーで0-100% | 高 |

**考慮事項:**

- 初回設定にかかる時間（離脱リスク）
- 設定を見直す頻度
- 「ちょっと好き」と「すごく好き」を区別する必要性
```

## 計算リソース

**目的**: 技術的な制約と許容範囲を確認する

### テンプレート

```markdown
### 質問: [処理]の実行タイミング

[処理]はリアルタイムで行いますか、バッチで行いますか？

**比較:**
| 方式 | レイテンシ | コスト | 鮮度 |
|------|-----------|--------|------|
| リアルタイム | [値] | [値] | [値] |
| バッチ | [値] | [値] | [値] |

**判断基準:**

- 許容レイテンシ: [値]
- コスト上限: [値]
- データ鮮度要件: [値]
```

#### 適用例

```markdown
### 質問: 推薦スコア計算のタイミング

推薦スコアはリアルタイムで計算しますか、バッチで計算しますか？

**比較:**
| 方式 | レイテンシ | コスト | 鮮度 |
|------|-----------|--------|------|
| リアルタイム | 100ms | 高（API呼び出し） | 最新 |
| バッチ（1時間） | 0ms（事前計算） | 低 | 1時間遅延 |
| ハイブリッド | 50ms | 中 | 準最新 |

**判断基準:**

- 許容レイテンシ: ページ表示は1秒以内？
- コスト上限: 月額いくらまで？
- データ鮮度要件: 「いいね」が即座に反映される必要は？
```

**判断基準:**

## 動的調整

**目的**: パラメータの固定/可変を決定する

### テンプレート

```markdown
### 質問: [パラメータ]の調整方法

[パラメータ]は固定値ですか、動的に変化しますか？

**選択肢:**

- A: 固定値（[値]）
- B: ルールベースで変動（[条件]で変化）
- C: 機械学習で自動調整

**考慮事項:**

- 変動させる根拠となるデータがあるか
- 予測不能な挙動のリスク
- デバッグの難易度
```

#### 適用例

```markdown
### 質問: 探索/活用比率の調整方法

推薦における「探索:活用」の比率は固定ですか、動的ですか？

**選択肢:**

- A: 固定値（例: 20:80）
- B: ユーザーの行動で変動（例: 新規ユーザーは探索多め）
- C: A/Bテストで最適値を探す

**考慮事項:**

- 固定なら実装がシンプル
- 動的なら「なぜその比率？」の説明が難しい
- 最初は固定、データが溜まったら動的化もあり
```

## 新規アイテム対応

**目的**: 新着コンテンツの扱いを決定する

### テンプレート

```markdown
### 質問: 新規[アイテム]の扱い

新しく追加された[アイテム]をどう扱いますか？

**選択肢:**

- A: 既存と同等に扱う（データ不足でも）
- B: 一定期間ブーストする（露出を増やす）
- C: 十分なデータが溜まるまで控えめに表示

**考慮事項:**

- 新着の価値（鮮度重視？）
- データ不足による推薦精度の低下
- クリエイターへの公平性
```

#### 適用例

```markdown
### 質問: 新着記事の推薦扱い

新しく投稿された記事をどう推薦に組み込みますか？

**選択肢:**

- A: 既存記事と同等に扱う（評価データなしで推薦）
- B: 24時間は「新着」として優先表示
- C: 最低10件の評価が付くまで推薦対象外

**考慮事項:**

- SCP財団は「新着を追いたい」ファンが多い？
- 評価データなしの記事を推薦するリスク
- 投稿者のモチベーション（すぐ見てもらいたい？）
```

## 質問選択ガイドライン

### ラウンド別の推奨カテゴリ

| ラウンド | 推奨カテゴリ                                                 | 目的               |
| -------- | ------------------------------------------------------------ | ------------------ |
| Round 1  | 定義の明確化、シグナルの優先順位、コールドスタート、成功指標 | 基本方針の確立     |
| Round 2  | 時間的変化、UIトレードオフ、属性の考慮                       | トレードオフの解決 |
| Round 3  | UIの粒度、計算リソース、段階的アプローチ                     | 技術制約の確認     |
| Round 4  | 動的調整、新規アイテム対応、MVP範囲確認                      | 最終確認           |

### 質問数の目安

- 1ラウンドあたり **最大3問**
- ユーザーの回答が明確なら **2問でも可**
- 深掘りが必要なら **同じカテゴリで追加質問**

### 質問のカスタマイズ

テンプレートはあくまで出発点。以下を考慮してカスタマイズ：

1. **ドメイン固有の用語に置き換え**: 「アイテム」→「記事」「商品」など
2. **具体的な選択肢を追加**: ユーザーの文脈に合わせた例
3. **不要な選択肢を削除**: 明らかに該当しないものは省略
4. **前の回答を踏まえた質問**: Round 1の回答を元にRound 2の質問を調整
