# 需求-测试追溯矩阵模板

## 编号规范

| 类型 | 前缀 | 格式 | 说明 |
|------|------|------|------|
| 功能点 | F | F-XXX | 用户可感知的独立功能 |
| 用户故事 | US | US-XXX | 用户使用场景 |
| 验收标准 | AC | AC-XXX | 可量化的完成条件 |
| 单元测试 | UT | UT-XXX | 验证内部逻辑 |
| 集成测试 | IT | IT-XXX | 验证组件协作 |
| E2E 测试 | E2E | E2E-XXX | 验证用户场景 |

## 矩阵格式

## 测试代码标注

### 方式一：注释标注

```typescript
describe('F-001: 用户注册', () => {
  describe('US-001: 使用邮箱注册', () => {
    // UT-001: AC-001 - 有效邮箱
    it('应该返回 true 当邮箱格式有效', () => {
      expect(validateEmail('test@example.com')).toBe(true);
    });

    // UT-002: AC-002 - 已存在邮箱
    it('应该抛出错误当邮箱已存在', () => {
      expect(() => register(existingEmail)).toThrow('邮箱已存在');
    });
  });
});
```

### 方式二：JSDoc 标注

```typescript
/**
 * @feature F-001
 * @story US-001
 */
describe('UserService.register', () => {
  /**
   * @testId UT-001
   * @acceptance AC-001
   */
  it('应该创建用户当邮箱有效', () => {
    // ...
  });
});
```

### 方式三：测试名称包含编号

```typescript
describe('validateEmail', () => {
  it('UT-001: AC-001 - 应该返回 true 当邮箱有效', () => {});
  it('UT-002: AC-002 - 应该返回 false 当邮箱无效', () => {});
});
```

## 自动化脚本

### 检查未覆盖的验收标准

```bash
#!/bin/bash
# check-ac-coverage.sh

# 从需求文档提取 AC 编号
AC_LIST=$(grep -oE "AC-[0-9]+" docs/devdocs/01-requirements.md | sort | uniq)

# 从测试代码提取已覆盖的 AC 编号
COVERED=$(grep -rhoE "AC-[0-9]+" tests/ --include="*.ts" | sort | uniq)

echo "未覆盖的验收标准:"
comm -23 <(echo "$AC_LIST") <(echo "$COVERED")
```

### 生成覆盖报告

```bash
#!/bin/bash
# generate-traceability-report.sh

echo "# 追溯覆盖报告"
echo ""
echo "| 类型 | 总数 | 已覆盖 | 覆盖率 |"
echo "|------|------|--------|--------|"

# 统计 F
F_TOTAL=$(grep -coE "F-[0-9]+" docs/devdocs/01-requirements.md)
F_COVERED=$(grep -rhoE "F-[0-9]+" tests/ --include="*.ts" | sort | uniq | wc -l)
echo "| 功能点 | $F_TOTAL | $F_COVERED | $((F_COVERED*100/F_TOTAL))% |"

# 统计 AC
AC_TOTAL=$(grep -coE "AC-[0-9]+" docs/devdocs/01-requirements.md)
AC_COVERED=$(grep -rhoE "AC-[0-9]+" tests/ --include="*.ts" | sort | uniq | wc -l)
echo "| 验收标准 | $AC_TOTAL | $AC_COVERED | $((AC_COVERED*100/AC_TOTAL))% |"
```

## 维护建议

1. **新增需求时**
   - 在需求文档中分配 F/US/AC 编号
   - 在追溯矩阵中添加条目，状态标记为 ❌

2. **编写测试时**
   - 在测试代码中添加 AC 编号标注
   - 更新追溯矩阵状态

3. **定期审查**
   - 运行覆盖检查脚本
   - 确保 P0 验收标准 100% 覆盖

4. **CI 集成**
   - P0 验收标准未覆盖时构建失败
   - 生成追溯覆盖报告
