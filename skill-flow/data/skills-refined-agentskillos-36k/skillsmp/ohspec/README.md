# OHSpec

AI 辅助需求分析与技术规范生成工具，自动将用户需求转化为结构化的 RFC 文档。

## 设计理念

AI 辅助需求分析面临的核心挑战：
- 复杂任务的上下文管理（代码扫描、分析、设计等信息累积）
- 大型代码库的高效扫描（平衡速度与覆盖率）
- 生成文档的质量稳定性（结构化、可审查、可追溯）
- 长时间会话的性能保持（避免上下文爆炸）

OHSpec 的设计选择：

**多 Agent 协作架构**
- 编排器负责协调，专家子代理负责执行
- 子代理在独立上下文中工作，返回结构化摘要
- 主上下文保持清洁，节省 19k-59k tokens

**外部记忆管理**
- 三文件架构（findings/progress/rfc）作为"外部记忆"
- 选择性读取，按需加载，支持长时间会话
- 可追溯决策历史，支持中断恢复

**智能扫描策略**
- ASSESS 阶段预评估代码库规模
- 动态选择扫描策略（直接/过滤/并行/摘要）
- 平衡扫描速度与结果质量

**多层质量门禁**
- 自动预检（结构、覆盖、量化）+ 人工审查（专家评分）
- 强制标准（3 个文件引用、4 类场景、8 个 DFX 维度）
- 确保输出质量稳定可靠

## 核心特性

- **智能路由**：根据复杂度自动选择执行策略（快速通道/标准流程/完整流程）
- **专家协作**：调度多个专业子代理（需求分析师、架构设计师、质量审查员等）
- **竞品分析**：自动加载 Android/iOS/Windows 设计模式和最佳实践
- **质量门禁**：强制检查 DFX 约束、场景覆盖、结构完整性
- **性能优化**：Resume 模式减少 70% 上下文消耗，执行速度提升 2-3 倍

## 工作流程

```
用户需求
   ↓
初始化三文件（rfc/findings/progress）
   ↓
ASSESS 评估（工具/规模/扫描策略）
   ↓
Dispatcher 基线扫描（scan-of-record）
   ↓
复杂度路由
   ├─ SIMPLE   → 快速通道（analyze+design 合并）
   ├─ MEDIUM   → 标准流程（analyze → design → audit）
   └─ COMPLEX  → 完整流程（+ 每阶段确认 + spike 验证）
   ↓
precheck → audit（默认自动流转）
   ↓
RFC 输出（5 个章节，需求化设计）
   ↓
（可选）手动 export 派生机读件（digest/tasks）
```

## 快速开始

```bash
# 一键生成 RFC（YOLO 模式）
/ohspec "为音频服务增加 3D 音效开关"

# 分阶段执行（手动模式）
/ohspec:analyze "需求描述"
/ohspec:design RFC-20260119-143000
/ohspec:audit RFC-20260119-143000

# 手动导出派生机读件（不自动触发）
# - rfc.digest.json：供后续 Agent 结构化消费
# - tasks.json：可选任务清单（用于编码 Agent）
/ohspec:export RFC-20260119-143000
/ohspec:export RFC-20260119-143000 --tasks
```

## 使用模式

### YOLO 模式（推荐）
自动执行完整流程，5-20 分钟内生成完整 RFC。适合大多数场景。

### 手动模式
分阶段确认，精细控制。适合复杂/关键项目。

详见 [USAGE-GUIDE.md](USAGE-GUIDE.md)

## RFC 输出

生成的 RFC 包含 5 个章节（需求化设计）：
- §1 上下文 - 代码库现状、相关实现
- §2 需求 - 功能目标、依赖影响、约束
- §3 契约 - 契约表/对齐矩阵/场景规格
- §4 DFX 约束 - 8 个维度 + 测试矩阵
- §5 设计决策 - 技术选型、权衡分析

可选派生物（手动执行 `/ohspec:export` 生成）：
- `rfc.digest.json` - 机读摘要（不在 RFC 内重复维护 JSON/YAML）
- `tasks.json` - 面向后续编码 Agent 的任务清单（可选）

## 核心概念

### 多 Agent 协作机制

**编排器模式**：主代理协调，子代理执行。避免上下文污染。

```
编排器（Orchestrator）
   ├─ 启动 Dispatcher（代码扫描、复杂度评估）
   ├─ 启动需求分析师（理解需求、澄清歧义）
   ├─ 启动架构设计师（设计契约、定义接口）
   └─ 启动质量审查员（DFX 检查、质量门禁）
```

**核心原则**：
- 编排器只协调，不直接执行代码扫描（Task 不可用时允许最小化回退扫描并记录）
- 子代理返回 JSON 摘要（≤500 tokens）
- 繁重工作委托给子代理，节省主上下文 19k-59k tokens

**子代理输入契约**：
```markdown
## 原始需求
{user_requirement}

## 上下文包
- findings.json 摘要: {关键模块、文件、依赖}
- progress.json 状态: {当前阶段、已完成阶段}
- 前序输出: {上一阶段的关键决策}

## 当前任务
{本阶段具体任务}

## 验收标准
{输出必须满足的条件}
```

### 上下文管理

**三文件架构**（外部记忆）：
- `findings.json` - 代码扫描结果、用户决策、技术约束
- `progress.json` - 执行时间线、任务状态、交互记录
- `rfc.md` - 最终交付物（增量生成）

**核心规则**：
- **2-Action Rule**：每 2 次代码扫描后必须更新 findings.json
- **Read Before Decide**：每个专家启动前必须重读 findings.json 和 progress.json
- **5-Question Reboot Test**：每个阶段开始前验证上下文充分性
- **3-Strike Error Protocol**：防止重复相同失败操作

**Token 预算管理**（三级预警）：
- 🟡 50% - 归档旧 RFC，压缩详细扫描结果
- 🟠 70% - 只保留当前 RFC 上下文，创建检查点
- 🔴 85% - 切换最小上下文模式，建议新会话

### 复杂度路由

- SIMPLE（单文件，<50行）→ 快速通道，5-8 分钟
- MEDIUM（多文件，单子系统）→ 标准流程，15-20 分钟
- COMPLEX（跨子系统）→ 完整流程，1-2 小时

## 质量保障

- ✅ 强制基线扫描（scan-of-record）：key_files ≥ 3（覆盖面达标）+ facts（项目事实+锚点）
- ✅ 选项式提问（避免开放式问题）
- ✅ DFX 量化检查（禁止模糊描述）
- ✅ 场景覆盖检查（正常/异常/边界/不支持）
- ✅ 对齐与变更矩阵（existing_ref 或 new_change_plan）
- ✅ 依赖影响矩阵 + 测试矩阵
- ✅ 自动预检（precheck）+ 人工审查（audit）

## 技术亮点

### 1. 编排器模式（Orchestrator Pattern）

**问题**：单一 Agent 处理复杂任务时，上下文会包含大量代码扫描结果、中间分析、设计草稿等，导致：

- 上下文爆炸（>100k tokens）
- 后续阶段性能下降（需要处理大量无关信息）
- 成本高昂（每次 API 调用都传输完整上下文）

**解决方案**：
- 主代理（编排器）只负责协调，不执行繁重任务
- 子代理在独立上下文中执行（代码扫描、需求分析、方案设计）
- 子代理返回 JSON 摘要（≤500 tokens），而非完整扫描结果
- 详细结果写入外部文件（findings.json），按需读取

**效果**：
- 主上下文节省 19k-59k tokens（约 70-85%）
- 执行速度提升 2-3 倍（Resume 模式）
- 成本降低 60-80%

### 2. 外部记忆管理（Manus Mode）

**问题**：传统 AI 工具依赖对话历史作为"记忆"，但对话历史会线性增长，无法选择性遗忘。

**解决方案**：
- **三文件架构**：findings.json（扫描结果）、progress.json（执行状态）、rfc.md（交付物）
- **选择性读取**：每个阶段只读取需要的部分（如设计阶段只读 findings 摘要）
- **增量写入**：RFC 按章节增量生成，避免重复传输
- **压缩策略**：Token 预算超过 50% 时自动归档历史数据

**效果**：

- 支持长时间会话而不爆上下文
- 可恢复中断的任务（Resume 模式）
- 可追溯决策历史（progress.json 记录所有交互）

### 3. 智能扫描策略（ASSESS 阶段）

**问题**：大型代码库（>10k 文件）直接扫描会导致：
- 超时（Grep 扫描 >5 分钟）
- 内存溢出（读取大量文件）
- 结果过多（>1000 个匹配）

**解决方案**：
- **预评估**：先用 Glob 统计文件数，Grep 预览匹配数
- **动态策略**：
  - 直接扫描（<100 文件）
  - 过滤扫描（100-1000 文件，限定目录）
  - 并行扫描（>1000 文件，分模块）
  - 摘要模式（>5000 文件，只扫描关键模块）
- **自适应**：根据实际匹配数动态调整策略

**效果**：
- 大型代码库扫描时间下降
- 避免超时和内存溢出
- 结果更聚焦（只返回相关文件）

### 4. Resume 模式（性能优化）

**问题**：每次启动子代理都需要加载 persona（~3,700 tokens），多阶段任务重复加载浪费资源。

**解决方案**：
- 使用 Task 工具的 `resume` 参数复用子代理
- 单个子代理连续执行多个阶段（analyze → design）
- 避免重复加载 persona 和上下文

**效果**：
- 简单任务：从 ~11,200 tokens 降至 ~3,800 tokens（节省 66%）
- 执行速度提升 2-3 倍
- 适用于 SIMPLE 和 MEDIUM 任务

### 5. 质量门禁体系

**问题**：AI 生成的文档质量不稳定，常见问题：
- 需求理解偏差
- 接口定义不完整
- DFX 约束模糊（"性能好"、"延迟低"）
- 场景覆盖不全（只考虑正常流程）

**解决方案**：
- **自动预检**（precheck）：结构完整性、场景覆盖、DFX 清单
- **人工审查**（audit）：质量审查员专家评分（0-100）
- **强制标准**：
  - 至少 3 个相关文件引用且覆盖面达标
  - 4 类场景覆盖（正常/异常/边界/不支持）
  - 8 个 DFX 维度量化（性能/安全/可靠性等）
  - 禁止模糊描述（自动检测）

**效果**：
- RFC 质量稳定（评分 >85 分）
- 减少人工修改工作量（从 >2 小时降至 <30 分钟）
- 可直接用于团队评审

## 已知不足

### 1. 成本较高
- 多次 API 调用（Dispatcher + 分析 + 设计 + 审查）
- 简单任务约 4-6 次调用，复杂任务 >10 次
- 缓解：Resume 模式减少 40-60% 调用次数

### 2. 复杂度评估可能不准确
- 依赖 Dispatcher 的启发式规则（文件数、行数、依赖数）
- 可能误判（如单文件但逻辑复杂）
- 缓解：支持手动指定复杂度（`/ohspec:analyze --complexity=COMPLEX`）

### 3. 依赖外部文件管理
- 需要读写 `.ohspec/rfcs/` 目录（兼容：`.claude/ohspec/rfcs/`）
- 文件损坏会导致流程中断
- 缓解：自动备份（checkpoint）、错误恢复机制

### 4. 竞品分析覆盖有限
- 当前仅支持 Android/iOS/Windows
- 知识库需要手动维护
- 缓解：支持用户自定义知识库路径

### 5. 无法处理极端场景
- 超大型代码库（>50k 文件）仍可能超时
- 极度复杂的需求（>10 个子系统）可能需要拆分
- 缓解：提供拆分建议、分阶段执行

## 未来演进方向

### 短期（3-6 个月）

**1. 智能复杂度评估**
- 引入机器学习模型（基于历史 RFC 数据）
- 考虑更多维度（代码复杂度、团队熟悉度、技术栈）
- 准确率提升至 >90%

**2. 上下文压缩优化**
- 语义检索代替全文搜索
- Token 消耗降低 50%

### 中期（6-12 个月）

**3. 动态专家组合**

- 根据需求自动选择专家（不仅限于预定义的 4 个核心专家）
- 支持用户自定义专家（如"前端专家"、"数据库专家"）
- 专家市场（社区贡献专家 persona）

**4. 增量更新机制**

- 支持修改需求后增量更新 RFC（而非重新生成）
- 变更影响分析（修改需求后自动标记受影响的章节）
- 版本对比（RFC v1 vs v2 diff）

### 长期（12+ 个月）

**5. 自适应学习**

- 从用户反馈中学习（RFC 被修改的部分 → 改进生成策略）
- 项目级学习（记住项目的编码规范、设计模式）
- 团队级学习（记住团队的偏好和约定）

**6. 端到端自动化**

- RFC → 代码生成（自动实现）
- RFC → 测试用例生成（自动测试）
- RFC → 文档生成（API 文档、用户手册）

**7. 协作增强**

- 多人协作（多个用户同时编辑 RFC）
- 评审工作流（RFC 审批、评论、修订）
- 与项目管理工具集成（Jira、GitHub Issues）

## 业界对比

| 维度 | 传统 AI 工具 | OHSpec | 业界前沿 |
|------|-------------|--------|----------|
| 上下文管理 | 对话历史（线性增长） | 外部记忆（三文件架构） | 向量数据库 + RAG |
| Agent 架构 | 单一 Agent | 编排器 + 多专家 | Multi-Agent System + 工具调用 |
| 质量保障 | 无 | 多层门禁（precheck + audit） | 形式化验证 + 测试生成 |

**OHSpec 的定位**：
- 已实现业界前沿的多 Agent 协作和外部记忆管理
- 主要面向需要设计文档的工作流，并给后续的代码自动开发生成足够的信息
- 未来演进方向与业界趋势一致（多Agent自动决策规划、超长上下文管理、超长时任务批量处理等）
