# TDD Red Stage Test Generation

You are implementing Test-Driven Development for the Scenario ID in the specified feature file:

$ARGUMENTS

Based on the provided feature file and application architecture, write FAILING tests first following strict TDD Red-Green-Refactor cycle.

## CRITICAL: MINIMAL TEST PRINCIPLE

**ONLY test what is EXPLICITLY stated in the scenario. DO NOT add:**
- Extra features or behaviors not mentioned in the scenario
- Complex state management beyond what's required
- Additional error cases not specified
- Performance optimizations not requested
- UI/UX enhancements not described
- Multiple variations of the same test

**If the scenario says "button is disabled" - test ONLY that the button is disabled.**
**If the scenario says "show error message X" - test ONLY for that exact message.**

## PRE-IMPLEMENTATION DISCOVERY (Required First Step)

**BEFORE writing any tests, you MUST search for existing code and tests.**

### Discovery Searches Required:

1. **Search for Existing Tests**
   - Search by scenario ID (e.g., `US001S02`, `US001-S02`, `US001_S02`)
   - Search by feature keywords from the scenario title
   - Search test directories for related test files
   - Check for skipped/disabled/pending tests that match this scenario

2. **Search for Existing Implementation Code**
   - Search for services, models, or components mentioned in the scenario
   - Search for functions/methods that might already implement this behavior
   - Check for partial implementations or TODOs related to this feature

3. **Classify the Situation**

| Existing Tests? | Existing Code? | Action |
|-----------------|----------------|--------|
| None | None | ✅ Proceed with new tests (standard RED phase) |
| Found (passing) | Found (working) | ⚠️ STOP - Feature may already be implemented. Verify coverage. |
| Found (failing/skipped) | None | ✅ Review existing tests, extend if needed, then proceed |
| Found (failing/skipped) | Partial | ⚠️ STOP - Analyze what exists. May need GREEN phase instead. |
| None | Partial | ⚠️ STOP - Write tests for existing code first, then extend |

### Mandatory Stop Triggers (ASK before proceeding):

- **Found passing tests** with same scenario ID → Ask if tests are sufficient
- **Found multiple test files** that could contain this scenario → Ask which to use
- **Found implementation code** that seems to do what scenario describes → Ask if it's correct
- **Found skipped/disabled tests** → Ask why they were disabled before re-enabling

## RED Phase - Write Minimal Tests at THREE Levels:

### 1. INTEGRATION TEST (End-to-End Scenario)

**Write ONE test that validates the EXACT scenario:**
- Test method name should match the scenario ID (e.g., `testUS001S02_PreventCaptureWhenQualityBelowThreshold`)
- Implement ONLY the Given-When-Then steps as written
- Assert ONLY what the scenario explicitly states
- Use real UI elements where the scenario mentions them

### 2. COMPONENT TEST (ViewModel/Controller Level)

**Write minimal tests for component behavior:**
- Test ONLY the state changes mentioned in the scenario
- One test per "Then" statement if they represent different behaviors
- Use existing mocks where available
- Don't create new properties/methods unless scenario requires them

### 3. UNIT TESTS (Service/Logic Level)

**Write focused tests for core logic ONLY if scenario specifies calculations:**
- Test specific thresholds mentioned (e.g., "below 300 lux")
- Test specific values mentioned (e.g., "quality score below 0.6")
- One test per explicit requirement
- Reuse existing models/enums where possible

## Key Requirements:

- **MINIMAL TESTS** - Write the absolute minimum to validate the scenario
- **EXACT REQUIREMENTS** - Test ONLY what's explicitly stated
- **REUSE EXISTING CODE** - Don't create new models/services if existing ones work
- **SIMPLE ASSERTIONS** - One assertion per requirement
- **NO OVER-ENGINEERING** - No extra features, no complex architectures
- **CLEAR MAPPING** - Each test maps to one scenario requirement
- **NO DUPLICATE FILES** - Never create a new test file if one already exists for this feature
- **EXTEND, DON'T DUPLICATE** - Add tests to existing test files rather than creating parallel files
- **DISCOVERY FIRST** - Always complete the discovery report before writing any test code

## Example for "button is disabled when quality is poor":

**WRONG - Over-engineered:**
- Testing multiple severity levels
- Testing animation states
- Testing complex state machines
- Creating new enums for every possible state

**RIGHT - Minimal:**
- Test: When quality < threshold, button.isEnabled == false
- That's it. One test, one assertion.

## Code Archaeology Reference (Optional)

When you find existing code during discovery, use these signals to interpret intent:

| Signal Found | Likely Intent | Action |
|--------------|---------------|--------|
| Passing tests for scenario | Feature implemented | Verify coverage, don't duplicate |
| Skipped/disabled tests | Work in progress or blocked | Ask why before proceeding |
| `TODO` or `FIXME` comments | Incomplete implementation | Read context, may inform approach |
| Feature flag wrapping | Experimental/unreleased | Check if flag should be enabled |
| Deprecated markers | Being replaced | Find the replacement |
| Empty stub methods | Placeholder for future | May be your starting point |
| Commented-out code | Previous attempt | Review for lessons learned |
