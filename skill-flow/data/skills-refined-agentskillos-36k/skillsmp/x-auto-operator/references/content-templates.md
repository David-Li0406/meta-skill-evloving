## 二、技术推文模板

### 2.1 技巧分享型

**模板结构**:
```
[Hook: 引人注目的开头]

[问题/痛点]

[解决方案: 1-3个要点]

[Call to Action]
```

**示例1: 编程技巧**
```
刚发现一个 Python 技巧，后悔没早点知道 🐍

问题: 字典合并写得又臭又长

以前:
dict1.update(dict2)

现在 (Python 3.9+):
merged = dict1 | dict2

一行搞定，还不改变原字典。

你们还有什么冷门但好用的技巧？
```

**示例2: LLM使用技巧**
```
用了3个月 Claude，总结出这个 Prompt 框架:

STAR 结构:
- Situation: 描述场景
- Task: 明确任务
- Action: 期望的行为
- Result: 输出格式

比 "你是一个专家..." 有效10倍。

详细教程见评论 👇
```

### 2.2 工具推荐型

**模板结构**:
```
[吸引标题]

[工具名称 + 一句话描述]

为什么推荐:
• 优点1
• 优点2
• 优点3

[适用场景]

[链接/获取方式]
```

**示例**:
```
又发现一个宝藏开源项目 ⭐

LlamaIndex: 让 LLM 理解你的数据

为什么我天天用:
• 5行代码接入任何数据源
• 自带 RAG 最佳实践
• 支持所有主流 LLM

适合: 想快速搭建知识库问答系统的开发者

GitHub 链接见评论
```

### 2.3 经验教训型

**模板结构**:
```
[承认错误/分享教训]

[背景]

[犯的错误]

[学到的教训]

[建议]
```

**示例**:
```
踩了一个月的坑，分享出来省得你们再踩

背景: 用 RAG 做客服系统

犯的错: 直接用默认的 chunk_size=1000

结果: 检索结果乱七八糟，回答驴唇不对马嘴

教训: 不同类型的文档需要不同的切分策略
- FAQ: 按问题切分
- 手册: 按段落切分
- 代码: 按函数切分

希望能帮到正在做 RAG 的朋友
```

### 2.4 对比评测型

**模板结构**:
```
[评测标题]

[评测背景/方法]

[结果对比表格或要点]

[结论和建议]
```

**示例**:
```
GPT-4 vs Claude 3 Opus: 代码生成实测对比

测试方法: 50个真实编程任务

结果:
✅ 算法题: GPT-4 略胜
✅ 长代码理解: Claude 更强
✅ Debug能力: 旗鼓相当
✅ 解释能力: Claude 更清晰

我的选择: 写代码用GPT-4，改bug用Claude

你们的体验呢？
```

## 三、Thread 长文模板

### 3.1 教程型 Thread

**结构**:
```
1/N [标题 + 预告]
2/N [为什么需要这个/痛点]
3/N [核心概念1]
4/N [核心概念2]
5/N [实战步骤1]
6/N [实战步骤2]
7/N [常见问题/坑]
N/N [总结 + CTA]
```

**示例: RAG 实战教程**
```
1/8 从零搭建 RAG 系统，我踩过的所有坑和最佳实践

花了2个月把公司知识库做成了 AI 问答系统，效果比预期好太多。

把完整方案分享出来，Thread 较长，建议收藏 🧵👇

---

2/8 先说为什么需要 RAG

LLM 的问题:
❌ 不知道你公司的信息
❌ 知识有截止日期
❌ 容易幻觉

RAG 解决方案:
✅ 让 LLM 先检索相关文档
✅ 基于文档生成回答
✅ 有据可查，减少幻觉

---

3/8 RAG 的核心流程

用户提问 → 向量检索 → 获取相关文档 → 拼接 Prompt → LLM 生成答案

看起来简单，魔鬼在细节...

---

4/8 第一个坑: Chunking 策略

❌ 错误做法: 固定 1000 字符切分

✅ 正确做法:
• FAQ → 按问答对切分
• 技术文档 → 按段落+标题切分
• 代码 → 按函数/类切分

不同文档类型需要不同策略!

---

5/8 第二个坑: Embedding 选择

测试了主流方案:
• OpenAI text-embedding-3-small: 性价比高
• BGE-M3: 开源最强，多语言
• Cohere embed-v3: 企业级首选

建议: 先用 OpenAI 跑通，再换开源

---

6/8 第三个坑: 检索策略

仅靠向量相似度不够！

我的组合拳:
1. 向量检索 top-20
2. BM25 关键词检索 top-20
3. Rerank 模型重排序
4. 取 top-5 给 LLM

准确率提升 40%+

---

7/8 第四个坑: Prompt 模板

别用默认 Prompt!

我优化后的模板:
"""
基于以下参考文档回答问题。
如果文档中没有相关信息，请明确说明。
不要编造信息。

参考文档:
{context}

问题: {question}
"""

---

8/8 总结

RAG 成功的关键:
1. 文档切分要合理
2. 检索要混合多种方法
3. Prompt 要明确约束
4. 持续根据bad case优化

完整代码我放在 GitHub 了，链接见评论。

觉得有用就转发一下，让更多人看到 🙏
```

## 四、图片推文模板

### 4.1 代码截图

**最佳实践**:
- 使用代码高亮工具 (Carbon, Ray.so)
- 深色主题更吸引眼球
- 代码行数控制在 15 行以内
- 关键部分加注释

**配文模板**:
```
[一句话描述这段代码解决什么问题]

[代码截图]

[使用场景或注意事项]
```

### 4.2 架构图/流程图

**最佳实践**:
- 使用 Excalidraw/Figma 制作
- 手绘风格更有亲和力
- 颜色不要超过 4 种
- 文字要清晰可读

**配文模板**:
```
[系统/流程名称] 架构一图流

[简要说明]

[想讨论的点或问题]
```

### 4.3 数据可视化

**最佳实践**:
- 图表要有明确的标题
- 数据来源要标注
- 关键数据点高亮
- 加上你的解读

## 九、AI 内容生成提示词

### 9.1 技术推文生成

```
你是一位专业的 AI/ML 技术博主。请生成一条关于 {topic} 的推文。

要求:
1. 开头要有吸引力 (Hook)
2. 内容要有实际价值
3. 语气专业但不枯燥
4. 长度控制在 200-280 字符
5. 结尾加一个互动问题
6. 不要使用过多 emoji (最多2个)

参考风格: {example_tweet}

请生成3个版本供选择。
```

### 9.2 Thread 生成

```
你是一位资深的 AI 技术专家和技术博主。
请为 "{topic}" 撰写一个 Thread (串推)。

要求:
1. 6-10条推文
2. 每条推文独立成段，但整体连贯
3. 第一条必须是强有力的 Hook
4. 包含具体的例子、数据或代码
5. 最后一条要有总结和 CTA
6. 每条推文约 200-280 字符
7. 技术深度适中，非专家也能理解

请按 "1/N ... 2/N ..." 的格式输出。
```

### 9.3 评论转推文

```
基于以下评论/笔记，扩展成一条完整的技术推文:

原始内容: {raw_content}

要求:
1. 保留核心观点
2. 增加具体例子
3. 添加吸引人的开头
4. 添加互动性的结尾
5. 长度适合发布

请生成推文。
```

## 十、内容检查清单

### 发布前必查

```
□ 内容是否 100% 技术相关？
□ 是否避免了敏感话题？
□ 是否有明显的错误/错别字？
□ 链接是否放在评论区？
□ 是否有吸引人的开头？
□ 是否有互动元素 (问题/CTA)？
□ 长度是否合适？
□ 图片/视频是否清晰？
□ 是否标注了信息来源 (如需)？
□ 发布时间是否合适？
```

### 质量评分标准

| 维度 | 优秀 | 良好 | 需改进 |
|------|------|------|--------|
| Hook | 让人忍不住继续看 | 基本吸引人 | 平淡无奇 |
| 价值 | 立刻可用 | 有参考价值 | 空洞无物 |
| 专业度 | 展示深度 | 基本准确 | 有明显错误 |
| 可读性 | 流畅易懂 | 基本清晰 | 晦涩难懂 |
| 互动性 | 激发讨论 | 引人思考 | 单向输出 |
