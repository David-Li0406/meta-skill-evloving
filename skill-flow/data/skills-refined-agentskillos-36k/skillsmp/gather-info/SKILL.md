# Gather Info Skill

タスクにおいて足りていない情報を深い質問で収集し、仕様ドキュメントとして保存する。

## 目的

1. タスクの曖昧さを検出し、明確化する質問を生成
2. 技術的な実装詳細、UI/UX設計、エッジケース、セキュリティ要件を網羅的に収集
3. 収集した情報を構造化されたMarkdownドキュメントとして保存

## タスクサイズの判定

### 大規模タスク (`<project>/docs/specs/` に保存)
以下のいずれかに該当する場合:
- 複数のコンポーネント/ファイルにまたがる変更
- 新機能の追加や大きなリファクタリング
- 外部サービスとの連携が必要
- データベーススキーマの変更を伴う
- 複数日にわたる作業が想定される
- ユーザーが「大きい」「複雑」「新機能」などのキーワードを使用

### 小規模タスク (Claude Code の Plan Mode `.claude/plans/` に保存)
以下のいずれかに該当する場合:
- 単一ファイルの修正
- バグ修正や小さな改善
- 既存機能の微調整
- 数時間以内に完了する作業
- ユーザーが「簡単」「小さい」「修正」などのキーワードを使用

判断に迷う場合はユーザーに確認する。

## 質問フロー

### Phase 1: コンテキスト把握 (必須)
1. タスクの背景と目的の確認
2. 期待する成果物の明確化
3. 優先度とスコープの確認

### Phase 2: 技術的詳細 (該当する場合)
1. アーキテクチャ選択
   - 使用する技術スタック
   - 既存コードとの整合性
   - パフォーマンス要件

2. データ構造
   - 入力/出力形式
   - データフロー
   - 状態管理

### Phase 3: UI/UX設計 (該当する場合)
1. ユーザーインターフェース
   - 画面レイアウト
   - インタラクション
   - レスポンシブ対応

2. ユーザー体験
   - ユーザーフロー
   - フィードバック表示
   - アクセシビリティ

### Phase 4: エッジケース・エラー処理 (必須)
1. 異常系の洗い出し
   - 無効な入力への対応
   - ネットワークエラー
   - 競合状態

2. エラーハンドリング
   - エラーメッセージ
   - リカバリー戦略
   - ログ出力

### Phase 5: セキュリティ・制約条件 (該当する場合)
1. セキュリティ要件
   - 認証/認可
   - データ保護
   - 入力検証

2. 制約条件
   - 依存関係
   - 互換性
   - ライセンス

## 質問の原則

### 深い質問のパターン
- 「なぜ」を3回深掘りする (5 Whys の応用)
- 具体例を求める
- 代替案を提示して選択を促す
- 暗黙の前提を明示化する

### 質問の優先順位
1. 実装を開始できない致命的な情報不足
2. 実装の方向性に影響する重要な詳細
3. 品質向上に寄与する追加情報

### 質問数の目安
- 1回の質問で3-4個の関連質問をグループ化 (AskUserQuestion の制限に合わせる)
- 回答を待ってから次のフェーズへ進む
- 全体で2-4ラウンドの質問を目安とする

## 質問形式

AskUserQuestion ツールを使用して質問する。各質問には:
- 明確な header (12文字以内)
- 具体的な options (2-4個の選択肢)
- 必要に応じて multiSelect: true

## 出力形式

### 質問フェーズ中の出力
```markdown
## 情報収集: [タスク名]

### 現在の理解
[現時点で把握している内容を箇条書き]

### 確認が必要な事項

#### [カテゴリ名]
1. [質問1]
   - 補足: [なぜこの情報が必要か]

2. [質問2]
   - 選択肢: A) [選択肢A] B) [選択肢B]
```

### 仕様ドキュメントの出力
質問が完了したら、タスクサイズに応じて以下の場所に仕様ファイルを作成:
- 大規模: `<project>/docs/specs/<task-name>-spec.md`
- 小規模: Plan Mode で使用される `.claude/plans/` 内のファイルに記載

テンプレートは `references/` ディレクトリを参照。

## 実行手順

1. タスクの説明を受け取る
2. 既存の関連コードやドキュメントを確認 (Glob, Grep, Read)
3. タスクサイズを判定し、出力先を決定
4. 不足情報を特定し、Phase 1から質問を開始
5. 回答を受けて次のPhaseへ進む (またはスキップ)
6. 全ての必要情報が揃ったら仕様ドキュメントを作成
7. ユーザーに仕様をレビューしてもらう

## 重要な注意事項

- 質問は明白な内容でないことを確認する
- 非常に深く掘り下げ、完了するまで継続的に質問を続ける
- 曖昧さを残さない
- ユーザーが「質問せずに進めて」と言った場合はスキップ
