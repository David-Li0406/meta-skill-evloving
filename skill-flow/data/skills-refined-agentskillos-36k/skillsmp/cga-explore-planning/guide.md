# cga-explore-planning 詳細ガイド

## 基本ルール

- Specが起点、無ければユーザーに確認
- src配下のコード変更は行わない
- `glossary.md` と `doc/decisions/` 必須確認
- コード調査のあと、必要があればGit履歴を調査
- 対話はメインで実施（透明性重視）
- TDD準拠の実装計画を作成

---

## Phase 0: 事前判定

`doc/plans/{issue-name}.md` が存在するか確認。計画済みなら `/cga-programming` へ誘導。

---

## Phase 1: コードベース調査

1. Specと `glossary.md`、`doc/decisions/` を読み込み
2. Explore でコードベース調査を実施

### Explore プロンプト例

**Explore（コードベース）**:
```
要件: {Spec}
調査: 関連実装、アーキテクチャ、類似パターン
出力: 要約のみ（関連ファイル、既存パターン、影響範囲）
```

---

## Phase 1.5: Git履歴調査の必要性判定

コード調査の結果を確認し、以下のいずれかに該当する場合はGit履歴調査を追加実施:

- 設計意図が不明（なぜこの実装になったのか？）
- 過去の変更経緯を知る必要がある
- 削除されたコードや機能の背景を知りたい
- 技術的負債の背景を理解する必要がある

**該当する場合**: Explore でGit履歴調査を追加実施

```
要件: {Spec}
調査: git log から設計判断の背景、避けるべきアプローチ
出力: 要約のみ（関連コミット、踏襲すべき方針、避けるべきアプローチ）
```

---

## Phase 2: 計画立案（メインで実施）

1. **選択肢の設計**: 各案の概要、メリット・デメリット、変更箇所
2. **トレードオフ分析**: 比較表作成（実装コスト、保守性、パフォーマンス、テスト容易性）
3. **ユーザーと対話**: AskUserQuestion で判断を仰ぐ
4. **実装計画作成**: TDDアプローチの実装順序を含める

### TDD実装順序

```markdown
## 実装計画（TDDアプローチ）

### Phase 1: テスト設計
- 受け入れ条件確認
- テストケース設計（ユニット + E2E）
- テストケースレビュー

### Phase 2: テスト実装（RED）
- ユニットテスト実装
- E2Eテスト実装
- テスト実行（RED確認）

### Phase 3: 最小実装（GREEN）
- ドメインモデル実装
- ユースケース実装
- コントローラー実装
- ルーティング設定
- テスト実行（GREEN確認）

### Phase 4: リファクタリング（REFACTOR）
- コード品質改善
- カバレッジ確認（80%以上）
```

---

## Phase 3: 結果の書き出し

`doc/plans/{issue-name}.md` に Write。以下の構成:

- 要件（Specより）
- 調査結果サマリ（ドメイン語彙、コードベース、外部情報）
- 設計決定（検討した選択肢、トレードオフ分析、最終決定）
- 実装計画（TDDアプローチ: テスト設計 → RED → GREEN → REFACTOR）
- 品質ゲート
- 次のアクション

---

## Phase 4: 次アクション

決定内容に応じて誘導:
- 重要な技術的判断 → `/cga-decision-record`
- 構造整理が先に必要 → `/cga-tf-refactoring`
- すぐ実装 → `/cga-programming`
